## 1. 执行摘要

DB-GPT v0.4.2 是一个基于 LLM 的数据交互框架，已具备基础的 NL-to-SQL、Dashboard 生成、Excel 分析等能力。然而，对标 Alibaba QuickBI 小Q 的四大支柱（问数、解读、洞察、报告），DB-GPT 当前整体能力覆盖率约为 **22%**，存在显著的产品化差距。

### 核心发现

| 维度 | 当前状态 | 差距总结 |
|---|---|---|
| 问数 (Q&A over Data) | ChatDB 有基础 NL-to-SQL，但缺乏数据准备、字段质量评估、知识库配置 | 覆盖率约 30%，核心缺失在数据治理与配置流程 |
| 解读 (Interpretation) | 仅有 Dashboard prompt 中的 thoughts 字段作为分析思路输出 | 覆盖率约 10%，几乎为零 |
| 洞察 (Insights) | 无自动化洞察发现能力 | 覆盖率 0% |
| 报告 (Reports) | ChatDashboard 可生成模板化图表组合，但不支持多章节叙事编排 | 覆盖率约 20%，MVT fin-report 正在补齐 |

### 战略判断

DB-GPT 的核心优势在于 **开源可控、模型无关、私有化部署**。QuickBI 小Q 的优势在于 **数据准备生态、企业级权限、阿里云集成**。建议采取差异化策略，在报告生成（MVT fin-report 方向）和解读引擎上建立独特竞争力，而非全面追赶 QuickBI 的数据准备管线。

## 2. 能力差距矩阵

### 2.1 总体覆盖率

| DB-GPT 模块 | QuickBI 支柱 | 当前覆盖率 | Phase 1 目标 | Phase 2 目标 | Phase 3 目标 |
|---|---|---|---|---|---|
| ChatExcel | 问数 (个人级) | 35% | 50% | 70% | 85% |
| ChatDB/ChatData | 问数 + 解读 | 25% | 50% | 70% | 85% |
| ChatBI | 问数 + 洞察 | 0% | 10% | 40% | 70% |
| ChatDashboard | 报告 | 20% | 40% | 65% | 80% |

### 2.2 ChatExcel 能力差距明细

**对标**: QuickBI 小Q 问数 -- 个人级数据分析

| 能力项 | QuickBI 小Q | DB-GPT 当前 | 差距 | 优先级 |
|---|---|---|---|---|
| 文件上传解析 | 支持 xlsx/csv/json | 支持 xlsx/csv，通过 ExcelReader + DuckDB 内存引擎 | 基本持平 | -- |
| 数据结构学习 | 自动字段类型推断 + 学习加速 | ExcelLearning 预处理阶段，采样数据推断列结构 | 缺少学习加速机制 | P1 |
| 字段质量评估 | 空值率/异常值/重复率评估 | 无 | **完全缺失** | P1 |
| 多表关联分析 | 支持多文件 join | 仅单文件分析 | **完全缺失** | P2 |
| 图表类型丰富度 | 20+ 图表类型 | 8 种 AntV 图表 (line/pie/table/scatter/bubble/donut/area/heatmap) | 基本够用 | P2 |
| 智能图表推荐 | 基于数据特征自动推荐 | Prompt 中引导 LLM 选择，无数据驱动的推荐逻辑 | 体验差距 | P1 |
| 自然语言交互 | 多轮对话 + 上下文 | 支持（BaseChat 的 history_message 机制） | 基本持平 | -- |

**数据流差距**:

当前 ChatExcel 的数据流为: `文件上传 -> ExcelReader -> ExcelLearning(采样学习) -> DuckDB -> NL-to-SQL -> 结果展示`

QuickBI 小Q 的数据流为: `文件上传 -> 字段质量评估 -> 学习加速(加速元数据推断) -> 数据集配置 -> 知识库(业务逻辑+正则) -> 权限管理 -> NL 查询 -> 校验 -> 准确结果`

DB-GPT 缺失的环节: **字段质量评估**、**学习加速**、**知识库配置**、**查询校验**。

### 2.3 ChatDB/ChatData 能力差距明细

**对标**: QuickBI 小Q 问数 + 解读

| 能力项 | QuickBI 小Q | DB-GPT 当前 | 差距 | 优先级 |
|---|---|---|---|---|
| 数据库连接 | 多数据源支持 | 通过 LOCAL_DB_MANAGE 支持多数据库连接 | 基本持平 | -- |
| 表发现机制 | 向量检索 + 元数据管理 | DBSummaryClient 向量检索表信息 | 基本持平 | -- |
| NL-to-SQL 生成 | 高准确率 SQL 生成 | ChatWithDbAutoExecute，基于 Prompt + 表结构生成 | 准确率待验证 | P0 |
| SQL 校验与纠错 | 语法校验 + 执行计划检查 + 自动纠错 | 仅 Prompt 约束（"仔细检查 SQL 正确性"） | **严重缺失** | P0 |
| 数据集配置 | 单表/多表 join/聚合指标 | 无数据集概念，直接查表 | **完全缺失** | P0 |
| 业务知识库 | 支持业务逻辑 + 正则匹配 | ChatKnowledge 模块独立，未与 ChatDB 集成 | 集成缺失 | P0 |
| 查询结果解释 | 一键智能解读 | 无（仅返回 thoughts 摘要） | **完全缺失** | P1 |
| 异常检测 | 自动检测数据异常并高亮 | 无 | **完全缺失** | P1 |
| 趋势分析 | 趋势分析 + 预测 | 无 | **完全缺失** | P2 |
| 根因分析 | 下钻式根因分析 | 无 | **完全缺失** | P2 |
| 权限管理 | 行列级权限控制 | 无 | **完全缺失** | P1 |

**用户体验差距**:

ChatDB 当前流程是: `选择数据库 -> 输入自然语言 -> 生成 SQL -> 执行 -> 返回结果 (thoughts + sql)`。用户无法在查询前配置数据范围、无法理解查询结果、无法下钻分析。

QuickBI 小Q 的引导式流程: `配置数据源 -> 字段质量评估 -> 定义业务指标 -> 知识库配置 -> 引导查询 -> 结果校验 -> 智能解读`。用户每一步都有引导和反馈。

### 2.4 ChatBI 能力差距明细

**对标**: QuickBI 小Q 问数 + 洞察

| 能力项 | QuickBI 小Q | DB-GPT 当前 | 差距 | 优先级 |
|---|---|---|---|---|
| 与现有 ChatBI 集成 | 不适用（QuickBI 本身就是 BI） | ChatScene 中未定义 ChatBI 场景 | **架构缺失** | P0 |
| 自动化洞察发现 | 自动发现数据模式与异常 | 无 | **完全缺失** | P1 |
| 跨维度模式识别 | 跨维度关联分析 | 无 | **完全缺失** | P2 |
| 主动洞察推送 | 基于数据变化主动推送洞察 | 无 | **完全缺失** | P2 |

### 2.5 ChatDashboard 能力差距明细

**对标**: QuickBI 小Q 报告

| 能力项 | QuickBI 小Q | DB-GPT 当前 | 差距 | 优先级 |
|---|---|---|---|---|
| 图表生成 | 丰富图表库 | 5 种基础图表 (Table/LineChart/BarChart/PieChart/IndicatorValue) | 基本够用 | -- |
| 多图表组合 | 自由组合 | Prompt 指定 4-8 个分析维度，自动生成多图表 | 基本可用 | -- |
| 数据驱动叙事 | 完整叙事报告 | 无叙事逻辑，图表独立生成 | **严重缺失** | P0 |
| 多章节编排 | 多章节报告组装 | 无章节概念（dashboard.json 仅为扁平图表列表） | **严重缺失** | P0 |
| 报告模板管理 | 模板市场 + 自定义 | 硬编码 dashboard.json 模板 | **体验差距** | P1 |
| 报告导出 | PDF/Word/PPT | 无 | **完全缺失** | P1 |
| 数据联动 | 图表间数据联动 | 无 | **完全缺失** | P2 |

**数据流差距**:

当前 ChatDashboard 的数据流: `选择数据库 -> 加载 dashboard.json 模板 -> NL 输入 -> LLM 生成多维度 SQL + 图表类型 -> DashboardDataLoader 执行 SQL -> 组装 ChartData -> 返回 ReportData`

QuickBI 小Q 报告的数据流: `选择数据源 -> 定义报告结构 -> 各章节独立生成(含叙事文本) -> 图表生成 -> 章节编排 -> 报告组装 -> 导出`

关键差距: DB-GPT 的 ChatDashboard **没有叙事文本生成能力**（仅 `thoughts` 字段作简要描述），**没有章节编排概念**，**没有报告导出能力**。

**注**: MVT fin-report 项目正在补齐多章节报告编排能力，当前已实现 4 章节 Markdown 输出（经营总论、经营总览、核心经营指标分析、事业部经营结构分析），这是 ChatDashboard 演进的重要基础。

## 3. 用户故事分解

### 3.1 Epic: ChatExcel 升级 -- 增强个人级数据分析

**Epic 描述**: 将 ChatExcel 从"上传文件 + 问问题"的基础体验，升级为"智能数据准备 + 引导式分析 + 结果解读"的完整个人分析体验。

#### Feature E1-F1: 数据质量评估

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E1-F1-US1 | 数据分析师 | 上传 Excel 后看到字段质量报告（空值率、异常值、重复率） | 在分析前了解数据质量，避免基于脏数据做决策 |
| E1-F1-US2 | 数据分析师 | 系统自动提示数据质量问题（如日期格式不一致、数值列含文本） | 快速修复数据问题，提升后续分析准确性 |
| E1-F1-US3 | 数据分析师 | 在质量报告中选择忽略/修复问题字段 | 控制分析的数据范围 |

**验收标准**:
- AC1: 上传文件后 10 秒内展示字段质量报告
- AC2: 每个字段显示: 空值率、数据类型推断结果、唯一值数量、异常值标记
- AC3: 用户可标记字段为"忽略"，后续分析自动跳过
- AC4: 质量报告支持中英文

**依赖**: 无外部模块依赖，可在 ExcelReader 层面独立实现

#### Feature E1-F2: 智能图表推荐引擎

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E1-F2-US1 | 数据分析师 | 系统根据查询结果自动推荐最佳图表类型 | 不需要手动选择图表，降低使用门槛 |
| E1-F2-US2 | 数据分析师 | 看到图表推荐的理由（如"检测到时间序列数据，推荐折线图"） | 理解推荐逻辑，建立信任 |

**验收标准**:
- AC1: 基于返回数据的特征（字段数、数据类型、记录数）自动推荐图表
- AC2: 推荐逻辑: 时间序列 -> 折线图/面积图, 分类对比 -> 柱状图, 占比分析 -> 饼图/环形图, 多维关系 -> 散点图/气泡图
- AC3: 用户可切换为其他图表类型

**依赖**: 需要 ChatExcel 的 output_parser 扩展

#### Feature E1-F3: 多表关联分析

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E1-F3-US1 | 数据分析师 | 同时上传多个 Excel 文件并定义关联关系 | 进行跨文件分析，无需手动合并数据 |
| E1-F3-US2 | 数据分析师 | 系统自动建议可能的关联字段 | 减少手动配置工作 |

**验收标准**:
- AC1: 支持上传 2-5 个文件
- AC2: 支持通过自然语言描述关联关系
- AC3: 自动检测相同名称的列并建议关联

**依赖**: ExcelReader 需支持多 DuckDB 表注册，NL-to-SQL 需支持多表 JOIN

### 3.2 Epic: ChatDB/ChatData 问数配置 -- 数据准备管线

**Epic 描述**: 为 ChatDB 增加企业级数据准备能力，使 NL-to-SQL 从"裸查询"升级为"有上下文的精准查询"。

#### Feature E2-F1: 数据集 (Dataset) 管理

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E2-F1-US1 | 数据管理员 | 定义数据集（单表/多表 JOIN/聚合指标） | 为不同分析场景提供标准化的数据入口 |
| E2-F1-US2 | 数据管理员 | 在数据集上定义计算字段和业务指标（如 GMV = 订单金额 - 退款金额） | LLM 生成的 SQL 能正确使用业务指标定义 |
| E2-F1-US3 | 业务用户 | 在查询时选择预定义的数据集而非直接选数据库表 | 降低查询复杂度，提升 SQL 准确率 |

**验收标准**:
- AC1: 数据集配置 UI 支持可视化定义 JOIN 关系
- AC2: 计算字段支持 SQL 表达式定义
- AC3: 数据集定义后自动注入 NL-to-SQL 的 Prompt 上下文
- AC4: 数据集配置持久化存储

**依赖**: 需要新的数据集管理模块（DatasetManager），与 DBSummaryClient 集成

#### Feature E2-F2: 业务知识库集成

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E2-F2-US1 | 数据管理员 | 配置业务术语表（如"活跃用户"定义为"近 30 天内有登录行为的用户"） | LLM 理解业务语言而非仅理解数据库字段名 |
| E2-F2-US2 | 数据管理员 | 配置常用查询模式（正则匹配 + 对应 SQL 模板） | 常见查询不依赖 LLM 生成，提升准确率和响应速度 |
| E2-F2-US3 | 业务用户 | 用业务术语提问而非必须使用数据库字段名 | 降低使用门槛 |

**验收标准**:
- AC1: 知识库支持 CRUD 操作
- AC2: NL-to-SQL 流程中优先匹配知识库规则，匹配失败再走 LLM 生成
- AC3: 知识库匹配率作为可追踪指标
- AC4: 支持导入导出知识库配置

**依赖**: 与 ChatKnowledge 模块集成，需要知识库检索 API

#### Feature E2-F3: SQL 校验与纠错引擎

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E2-F3-US1 | 业务用户 | 生成的 SQL 执行前自动校验语法和逻辑正确性 | 避免错误查询返回误导性结果 |
| E2-F3-US2 | 业务用户 | SQL 出错时系统自动尝试修复并解释修复内容 | 减少交互轮次，提升体验 |
| E2-F3-US3 | 系统管理员 | 查看 SQL 生成准确率统计 | 持续优化 Prompt 和知识库 |

**验收标准**:
- AC1: SQL 执行前通过语法检查（AST 解析）
- AC2: SQL 执行失败时，错误信息 + 原始 SQL 反馈给 LLM 进行自动纠错（最多重试 2 次）
- AC3: 记录每次 SQL 执行成功/失败状态，支持准确率报表
- AC4: 支持 SQL 执行超时保护（默认 30 秒）

**依赖**: 需要数据库方言适配层，与 BaseChat.do_action 集成

### 3.3 Epic: ChatDB/ChatData 数据解读 -- Interpretation Engine

**Epic 描述**: 为 ChatDB 增加智能解读能力，从"返回数据"升级为"解释数据"。

#### Feature E3-F1: 查询结果智能解读

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E3-F1-US1 | 业务用户 | 查询结果返回时附带自然语言解读（如"销售额环比增长 15%，主要由华东区域贡献"） | 无需自己分析数据，直接获得结论 |
| E3-F1-US2 | 业务用户 | 选择解读模式：简洁模式（基础统计）、详细模式（多维度分析）、智能模式（自动选择） | 根据场景获取不同深度的解读 |

**验收标准**:
- AC1: 每次查询结果附带解读文本（>100 字，非简单复述）
- AC2: 解读包含: 关键统计量（最大值/最小值/均值/趋势）、数据模式识别、异常标记
- AC3: 支持三种解读模式切换
- AC4: 解读响应时间 < 5 秒

**依赖**: 需要新的 InterpretationEngine 模块，调用 LLM 进行二次分析

#### Feature E3-F2: 异常检测与高亮

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E3-F2-US1 | 业务用户 | 数据中的异常值自动高亮并解释（如"3 月销售额异常下降 40%"） | 快速发现数据异常，不遗漏关键信号 |
| E3-F2-US2 | 业务用户 | 点击异常值可查看可能原因 | 快速定位问题根因 |

**验收标准**:
- AC1: 基于统计方法（Z-score / IQR）自动检测数值异常
- AC2: 基于同比/环比检测时间序列异常
- AC3: 异常数据在表格/图表中高亮显示
- AC4: 异常检测准确率 > 85%（通过标注数据集验证）

**依赖**: 需要统计计算引擎，与 ChartData 渲染层集成

#### Feature E3-F3: 趋势分析与预测

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E3-F3-US1 | 业务用户 | 对时间序列数据自动生成趋势分析（上升/下降/平稳 + 置信度） | 了解数据走向，辅助决策 |
| E3-F3-US2 | 业务用户 | 查看未来 3-6 期的趋势预测（附置信区间） | 提前规划 |

**验收标准**:
- AC1: 自动识别时间序列数据（检测日期/时间类型列）
- AC2: 生成趋势描述文本 + 趋势图（含预测区间）
- AC3: 预测使用简单统计方法（移动平均 / 线性回归），不要求深度学习
- AC4: 预测结果标注置信度

**依赖**: 需要时序分析模块

### 3.4 Epic: ChatBI 集成 -- 与现有 ChatBI 产品对接

**Epic 描述**: 建立 ChatBI 场景，将 DB-GPT 的 NL-to-SQL + 解读能力与现有 ChatBI 产品（如自研 BI 或第三方 BI）集成。

#### Feature E4-F1: ChatBI 场景定义

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E4-F1-US1 | 系统管理员 | 在 ChatScene 中注册 ChatBI 场景 | 用户可以在 BI 界面直接使用 NL 交互能力 |
| E4-F1-US2 | 业务用户 | 在 BI 看板中选择任意图表，点击"智能解读"按钮 | 无需切换场景即可获得数据解读 |

**验收标准**:
- AC1: 新增 ChatScene.ChatBI 枚举值
- AC2: ChatBI 继承 ChatWithDbAutoExecute 的 NL-to-SQL 能力
- AC3: ChatBI 支持接收 BI 看板的图表数据作为解读输入
- AC4: 提供标准 REST API 供 BI 前端调用

**依赖**: 需要与 BI 前端团队协调 API 接口

#### Feature E4-F2: BI 数据源适配

| User Story | 作为... | 我想要... | 以便... |
|---|---|---|---|
| E4-F2-US1 | 系统管理员 | 配置 ChatBI 对接的数据源（与 BI 共享数据源配置） | 避免重复配置，保持数据一致性 |

**验收标准**:

## 5. 竞争定位与差异化策略

### 5.1 DB-GPT 独特优势

| 优势维度 | DB-GPT | QuickBI 小Q |
|---|---|---|
| 开源可控 | 完全开源，可自行部署和定制 | 闭源 SaaS，受限于阿里云生态 |
| 模型无关 | 支持任意 LLM (OpenAI/通义/私有模型) | 绑定阿里通义千问 |
| 私有化部署 | 支持完全离线部署 | 需要阿里云环境 |
| 框架灵活性 | AWEL 工作流引擎，可自由编排 | 固定产品形态 |
| 多数据库支持 | MySQL/PostgreSQL/SQLite/DuckDB 等 | 以阿里云数据源为主 |
| 报告生成深度 | MVT fin-report 方向可实现深度业务报告 | 报告功能偏通用 |
| 定制化成本 | 低（开源框架，按需定制） | 高（需购买企业版+定制开发） |

### 5.2 QuickBI 小Q 的护城河

| 优势维度 | QuickBI 小Q | DB-GPT 差距 |
|---|---|---|
| 数据准备生态 | 完整的数据接入、清洗、加工管线 | DB-GPT 无数据 ETL 能力 |
| 用户基础 | 大量阿里云企业客户 | 开源社区用户为主 |
| BI 功能完整性 | 完整的 BI 套件（数据源->报表->仪表板->分享） | DB-GPT 仅有基础图表 |
| 权限与治理 | 成熟的行列级权限、审计、血缘 | 无权限管理 |
| 生态集成 | 阿里云全家桶集成 | 需自行集成 |
| 产品成熟度 | 迭代多年，功能完善 | v0.4.2 早期阶段 |

### 5.3 差异化策略建议

**核心策略: 深耕垂直场景 + 开源生态 + 私有化优势**

#### 策略一: 报告生成作为杀手级特性

QuickBI 小Q 的报告功能偏通用模板，而 DB-GPT 通过 MVT fin-report 方向可以实现:
- 深度业务报告（如月度经营分析报告，含多维度、多章节、数据驱动叙事）
- 可定制的报告模板（用户可定义章节结构、分析维度、数据映射）
- 离线报告生成（定时生成 + 邮件分发）

**差异化点**: QuickBI 做"通用报告"，DB-GPT 做"深度行业报告"。

#### 策略二: 模型自由度作为核心竞争力

企业对 LLM 选型的核心诉求:
- 数据安全（不希望数据出境）-> 需要私有模型
- 成本控制（避免按 token 付费）-> 需要本地模型
- 效果优化（不同场景用不同模型）-> 需要模型路由

DB-GPT 的 AWEL 工作流 + 模型路由能力是 QuickBI 无法提供的。

#### 策略三: 私有化部署打企业市场

金融、政务、军工等行业:
- 数据不能上云
- 需要私有化部署
- 需要信创适配（国产模型 + 国产数据库）

DB-GPT 的开源 + 私有化特性天然适配这些场景。

#### 策略四: 开发者生态构建

- 提供 SDK/API，让开发者基于 DB-GPT 构建自己的数据分析产品
- AWEL 工作流引擎作为平台级能力开放
- 插件市场（数据源插件、图表插件、分析算法插件）

### 5.4 自建 vs 集成 风险评估

| 方案 | 优势 | 风险 | 建议 |
|---|---|---|---|
| 完全自建 | 完全可控，深度定制 | 开发周期长（12+ 月），功能覆盖面有限 | Phase 1 核心能力自建 |
| 集成 QuickBI API | 快速上线，功能完善 | 依赖第三方，失去差异化，无法私有化 | 不推荐 |
| 混合方案 | 核心自建 + 非核心集成 | 集成复杂度，用户体验碎片 | 仅限非核心能力（如 BI 渲染） |
| 开源社区共建 | 分摊开发成本，加速迭代 | 质量控制难度，方向协调成本 | Phase 2+ 持续推进 |

**建议**: Phase 1 和 Phase 2 以自建为主，在 Phase 3 考虑引入开源社区力量共建洞察引擎等非核心模块。

### 5.5 关键风险与缓解措施

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| NL-to-SQL 准确率不达标 | 高 | 致命 | Phase 1 全力投入 SQL 校验 + 知识库，建立准确率基线 |
| LLM 成本过高 | 中 | 高 | 本地模型降级方案 + 结果缓存 + AWEL 模型路由 |
| QuickBI 快速迭代拉大差距 | 中 | 中 | 聚焦差异化场景（深度报告 + 私有化），不全面追赶 |
| 开源社区活跃度不足 | 中 | 中 | 算法层面建立技术壁垒（解读引擎、洞察引擎），不完全依赖社区 |
| 企业级功能缺失影响商用 | 高 | 高 | Phase 2 启动权限管理，Phase 3 完善 |
