## 1. 数据分析场景地图

### 1.1 分类维度定义

场景地图采用 **三维分类体系**，从数据对象粒度、分析自由度和用户角色三个维度交叉定义场景：

| 维度 | 含义 | 选取理由 |
|------|------|---------|
| **数据对象粒度** | 分析操作所作用的数据抽象层级 | 决定了系统的数据治理复杂度和元数据管理需求 |
| **分析自由度** | 终端用户执行分析任务时的自由程度/复杂度 | 决定了交互模式设计（预定义 vs 自由对话 vs 自动推送） |
| **用户角色** | 谁在使用这个场景 | 决定了产品形态、权限模型、学习曲线 |

#### 维度一：数据对象粒度

| 层级 | 数据对象 | 说明 | 典型特征 |
|------|---------|------|---------|
| L0 | **个人文件 (Personal File)** | 用户上传的 Excel/CSV 文件 | 数据规模小、Schema 自由、无治理要求、即用即弃 |
| L1 | **领域宽表/数据集 (Domain Dataset)** | 数据仓库中的主题宽表或数据集 | 有业务语义、经过 ETL 清洗、有明确 Owner |
| L2 | **指标 (Metric/Indicator)** | 预定义的业务度量（KPI、OKR 等） | 有计算公式、有维度、有口径定义、有目标值 |
| L3 | **报表/报告 (Report)** | 结构化的分析输出物 | 有章节结构、有叙事逻辑、有受众定义、有发布流程 |

从 L0 到 L3，数据治理程度递增、用户自由度递减、产品化程度递增。

#### 维度二：分析自由度（任务复杂度）

| 自由度等级 | 模式 | 用户行为 | 典型任务 |
|-----------|------|---------|---------|
| F0 | **固定查询** | 选择预定义模板，填入参数 | "查上月销售报表"、"看今日 GMV" |
| F1 | **引导式分析** | 系统引导用户逐步选择维度/度量 | "帮我分析各区域销售趋势" → 选择区域 → 选择时间范围 → 选择指标 |
| F2 | **自由探索** | 用自然语言自由提问 | "为什么华东区 Q3 利润下降？"、"哪个产品线的增速最快？" |
| F3 | **自动洞察** | 系统主动发现并推送 | "检测到华南区退货率异常上升 35%，可能原因：..." |

从 F0 到 F3，LLM 参与度递增、实现复杂度递增、对数据质量要求递增。

#### 维度三：用户角色

| 角色 | 核心诉求 | 技术能力 | 占比 |
|------|---------|---------|------|
| **业务决策者** | 快速获得结论、趋势、异常预警 | 不懂数据库/SQL | ~40% |
| **业务分析师** | 灵活探索数据、验证假设 | 懂业务、基础 SQL | ~35% |
| **数据工程师** | 配置数据源、治理数据质量 | 懂数据仓库/ETL | ~15% |
| **数据管理者** | 管理指标口径、审批报表 | 懂数据治理框架 | ~10% |

### 2.2 Layer 4: Agent Skills Market（智能分析技能市场）

#### 2.2.1 什么是 Agent Skills？

Agent Skills 是可复用的、面向特定分析场景的智能分析能力单元。每个 Skill 封装了：
- **触发条件**: 什么情况下激活这个 Skill
- **Prompt 模板**: 与 LLM 交互的指令模板
- **数据源配置**: Skill 作用的数据集/指标
- **输出格式**: Skill 返回的结构化结果

### 2.4 Layer 2: 黄金数据集（AI-Ready 数据资产层）

#### 2.4.1 什么是黄金数据集？

| 维度 | 普通数据表 | 黄金数据集 (AI-Ready) |
|------|-----------|---------------------|
| Schema 文档 | 可能缺失 | **完整中英文注释 + 业务语义描述** |
| 数据质量 | 无保障 | **空值率 <5%、格式统一、无重复** |
| 指标定义 | 分散在 SQL 脚本中 | **集中管理、有口径说明、有计算公式** |
| 维度层次 | 扁平 | **有层次结构、支持下钻** |
| 元数据 | 基础 | **含字段质量评分、使用频率、关联关系、敏感等级** |
| 访问控制 | 粗粒度 | **行列级权限、脱敏规则** |
| 更新机制 | 手动/ETL | **定时更新 + 数据新鲜度 SLA** |

#### 2.6.2 为什么是这个矩阵？（设计逻辑）

**原则 1: AI 参与度决定产品归属**

```
AI 参与度低（模板、配置、拖拽）  →  FineBI 主导
AI 参与度中（NL 交互、简单推理）  →  ChatBI 主导
AI 参与度高（深度推理、叙事生成）  →  DB-GPT 主导
```

**原则 2: 数据粒度决定技术复杂度**

```
个人文件（L0）  →  DuckDB 内存引擎足够，无需数据治理
领域数据集（L1） →  需要黄金数据集 + 元数据管理 + 权限控制
指标（L2）     →  需要指标管理系统 + 口径管理 + 归因引擎
报告（L3）     →  需要报告编排引擎 + 叙事生成 + 模板管理
```

**原则 3: Skills 是连接资产与产品的桥梁**

```
Skills = f(数据对象, 分析任务, 知识库) → 分析结果
         ↑           ↑           ↑
    Layer 2      用户输入     Layer 3
    (数据集)     (自由度)    (知识库)
```

| **实现路径** | 定时触发 → 洞察引擎发现要点 → 报告引擎组织叙事 → 生成报告 → 推送 |
| **所需模块** | InsightEngine (新建) + ReportOrchestrator (现有) + Scheduler (新建) |

## 4. 核心模块 L1/L2 分解

### 4.1 模块层级定义

| 层级 | 含义 | 说明 |
|------|------|------|
| **L1** | 必要模块 (Must-Have) | 没有这些模块，核心场景无法运行 |
| **L2** | 增强模块 (Should-Have) | 有了这些模块，体验和准确率显著提升 |
| **L3** | 锦上添花 (Nice-to-Have) | 完善体验，非核心路径 |

### 4.2 L1 必要模块（Phase 1 必须完成）

#### L1-01: SQL 校验纠错引擎 (SQLValidationEngine)

```
位置: pilot/capability/sql_validator/
优先级: P0
支撑场景: S-03, S-07, S-11, S-14, S-15

子模块:
├── sql_syntax_checker.py    # AST 语法校验（基于 sqlparse）
├── sql_execution_planner.py # 执行计划检查（EXPLAIN）
├── sql_auto_corrector.py    # LLM 驱动的自动纠错
└── sql_dialect_adapter.py   # 多方言适配（MySQL/PG/ClickHouse）

为什么必要:
当前 NL-to-SQL 准确率约 60-70%，无校验意味着错误 SQL 直接执行，
返回错误数据误导用户。SQL 校验是问数场景的底线能力。
```

#### L1-02: 业务知识库集成 (KnowledgeIntegration)

```
位置: pilot/capability/knowledge/
优先级: P0
支撑场景: S-07, S-08, S-10, S-11

子模块:
├── glossary_manager.py      # 业务术语表管理
├── sql_pattern_matcher.py   # SQL 模式匹配（正则 → SQL 模板）
├── metric_definition.py     # 指标口径定义
└── prompt_enricher.py       # 将知识库内容注入 NL-to-SQL Prompt

为什么必要:
ChatDB 当前直接查表，LLM 不理解业务语义。"活跃用户"对 LLM 来说只是
一个字段名，不知道它意味着"近30天有登录"。知识库是 NL-to-SQL 准确率
从 70% 提升到 85% 的关键杠杆。
```

#### L1-03: 数据集配置管理 (DatasetManager)

```
位置: pilot/capability/data_preparation/
优先级: P0
支撑场景: S-05, S-06, S-07, S-08

子模块:
├── datasource_manager.py    # 数据源注册与连接管理
├── dataset_manager.py       # 数据集（单表/多表/聚合）定义
├── field_manager.py         # 字段元数据管理（类型/语义/质量）
└── dataset_schema_builder.py # 为 NL-to-SQL Prompt 构建上下文

为什么必要:
当前 ChatDB 直接查数据库表，没有数据集抽象。用户需要知道表名和 JOIN
关系才能有效查询。数据集配置是降低用户认知负担、提升 SQL 准确率的
基础设施。同时是黄金数据集接入的入口。
```

#### L1-04: AnalysisChain 编排层

```
位置: pilot/capability/analysis_chain/
优先级: P0
支撑场景: S-08, S-10, S-14, S-15, S-16

子模块:
├── chain_orchestrator.py    # 链式编排引擎
├── analysis_stage.py        # 分析阶段抽象基类
├── stage_registry.py        # 阶段注册表
└── chain_executor.py        # 异步执行器（支持阶段间数据传递）

为什么必要:
BaseChat 是单次 LLM 调用生命周期，不支持链式推理。解读 → 归因 → 报告
等场景需要多阶段串联。AnalysisChain 是独立于 BaseChat 的编排层，被
BaseChat.do_action() 调用。不修改 BaseChat 内部结构。
```

#### L1-05: 报告编排引擎 (ReportOrchestrator)

```
位置: pilot/capability/report/
优先级: P0
支撑场景: S-13, S-14, S-15, S-16

子模块:
├── report_orchestrator.py   # 多章节编排引擎（基于 MVT chapter_writer）
├── chapter_generator.py     # 单章节生成（数据 + LLM 叙事）
├── report_template_mgr.py   # YAML 模板管理
└── report_renderer.py       # 输出渲染（Markdown → HTML/PDF）

为什么必要:
ChatDashboard 当前只是 ChartData 列表，无章节编排、无叙事生成。
MVT fin-report 已验证多章节报告可行，需要通用化。报告生成是 DB-GPT
的差异化杀手级特性。
```

### 4.3 L2 增强模块（Phase 2 完成）

#### L2-01: 解读引擎 (InterpretationEngine)

```
位置: pilot/capability/interpretation/
优先级: P1
支撑场景: S-07, S-08, S-11, S-12

子模块:
├── stat_summarizer.py       # 统计摘要生成（基础统计量）
├── anomaly_detector.py      # 异常检测（Z-score / IQR / 时序异常）
├── trend_analyzer.py        # 趋势分析（移动平均 / 线性回归）
├── root_cause_analyzer.py   # 根因分析（维度下钻 + LLM 推理）
└── interpretation_renderer.py # 解读结果渲染

为什么增强:
解读是从"返回数据"到"解释数据"的跨越。当前仅有 thoughts 字段作为
"思考摘要"，不是结构化解读。解读引擎是 DB-GPT 对标小Q解读能力的核心。
```

#### L2-02: 字段质量评估引擎 (FieldQualityEngine)

```
位置: pilot/capability/data_quality/

支撑场景: S-02, S-04

└── quality_suggester.py     # 质量改善建议

为什么增强:
数据质量是分析准确性的前提。上传脏数据分析，结果不可信。字段质量评估
是 QuickBI 问数配置的核心环节，也是 AI-Ready 数据集的前提。
```

#### L2-03: 指标管理 (MetricManager)

### 5.2 关键洞察总结

1. **DB-GPT 的核心战场是 F2+F3 自由度场景**：固定查询（F0）和简单引导（F1）是 FineBI 和 ChatBI 的主场，DB-GPT 不应分散精力追赶。NL 自由探索（F2）和自动洞察（F3）才是 LLM 的差异化价值所在。

2. **L1 模块是地基，缺一不可**：SQL 校验、知识库、数据集配置、编排层、报告引擎这 5 个 L1 模块构成了 DB-GPT 企业级能力的最小可行集。任何一个缺失都会导致多个场景无法支撑。

3. **黄金数据集是放大器**：同样一套 NL-to-SQL 引擎，接普通表准确率 60%，接黄金数据集可提升到 85%。数据治理的投资回报率高于算法优化。

4. **Skills Market 是连接器**：通过将能力封装为 Skills（问数 Skill、解读 Skill、报告 Skill），DB-GPT 可以被 ChatBI、小Q、甚至 FineBI 调用，成为真正的 Data Agent 底座。

5. **产品定位要清晰分工**：FineBI 做固定报表、ChatBI 做智能问数、DB-GPT 做深度分析和报告。避免重复建设，每个产品守住自己的核心场景。
