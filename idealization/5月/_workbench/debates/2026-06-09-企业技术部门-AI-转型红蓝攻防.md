# 专家辩论：企业技术部门 AI 转型红蓝攻防

## 议题

对 `ws-20260608-002` 和 `ds-20260608-002` 的核心判断做红蓝攻防：企业技术部门在 Agent / LLM 冲击下是否应定位为“企业智能能力提供方”，以及第一阶段是否应优先建设可信数据智能底座，而不是统一聊天入口。

## 关联 Session

- `ws-20260608-002`

## 辩论设定

- 目标：检验现有沉淀候选是否过于宏大、是否有可落地的能力边界、是否需要补充反例和验收指标。
- 成功标准：至少识别 3 条可保留判断、3 条弱假设、3 条后续验证问题，并给出是否补强现有候选的建议。
- 不进入正式知识库的内容：具体组织架构、FTE、预算、框架绑定结论，以及未经验证的企业内部采纳率判断。

## Round 1：立场陈述

### 产品经理

- Claim: 第一阶段不应包装成“企业 AI 战略平台”，而应落到一个业务用户能感知的可信问数和分析闭环。
- Evidence: `ws-20260608-002` 已把 MVP 收敛到“可问数据集 -> 语义指标 -> 问数/解读 -> 报告素材 -> 反馈评测 -> 资产沉淀”。
- Risk: 如果没有明确用户入口和反馈动作，底座建设会变成技术部门内部工程，业务侧只看到另一个复杂平台。
- Question: 第一阶段样板用户是谁，日常任务是什么，什么结果会让他们愿意重复使用？
- Recommendation: 保留“可信数据智能底座”判断，但在沉淀候选中补充用户可感知的验收指标。

### 战略专家

- Claim: “企业智能能力提供方”是可保留的定位，但它必须表达为组织能力资产，而不是技术部门权责扩张口号。
- Evidence: 现有知识库中大数据团队 AI 时代重新定义、Data Agent 能力分层、A1-A13 能力树都指向资产化、语义化、治理化。
- Risk: 如果组织考核仍按项目交付和需求响应计算，技术部门很难真正转向能力资产运营。
- Question: 哪些能力资产应该由技术部门强制统一，哪些应允许业务团队自治？
- Recommendation: 入库时标注这是战略定位和阶段性框架，不写成已经验证的组织事实。

### 架构专家

- Claim: 平台层优先级应从 Agent Runtime 下调到语义、权限、工具边界、Run Trace 和 Eval。
- Evidence: `ds-20260608-002` 已列出 P0 能力：可问数据集、指标语义、权限治理、SQL / Tool Guard、运行留痕、黄金问题集。
- Risk: 过早建设统一 Agent Runtime 会把团队拖入编排框架选型，而不是解决企业数据可信和可审计问题。
- Question: 现有 BI、数据平台、知识库和工具系统中，哪些能力能复用为底座，哪些需要新建？
- Recommendation: 把“统一平台”改写为“统一语义、治理、评测底座 + 多应用适配”的可降级架构。

### 产品专家

- Claim: 样板场景必须覆盖完整生命周期，而不是只证明模型能回答问题。
- Evidence: session 中已经强调反馈、评测和资产沉淀；这比一次性 ChatBI 演示更接近企业长期价值。
- Risk: 如果样板场景太宽，团队会把精力消耗在接入系统和权限协调上，无法证明单点价值。
- Question: 样板场景是否应限定在一个高频分析域，例如经营指标复盘、销售漏斗诊断或成本异常归因？
- Recommendation: 沉淀候选应补充“样板场景选择标准”，而不是只写平台能力清单。

### 红队专家

- Claim: 当前候选最大问题是“正确但泛”，缺少反证条件和失败判据。
- Evidence: “组织智能能力提供方”“能力资产”“Agentic Platform”都容易被认可，但也容易落成 PPT 和 Demo。
- Risk: 技术部门可能用新名词包装旧工作流：报表换成问答、数据目录换成知识中心、接口平台换成 Tool Hub。
- Question: 如果 3 个月后业务仍然绕开平台用个人 AI 工具，如何判定本路线失败并调整？
- Recommendation: 必须加入失败判据：无复用、无评测、无反馈、无权限审计时，不得宣称完成 AI 转型。

### 知识编辑专家

- Claim: 现有候选可以进入人工审阅，但需要补一层“边界和验收”后再考虑 approve。
- Evidence: 候选已有稳定判断、概念定义、建设优先级和决策记录候选，结构完整。
- Risk: 如果不补边界，正式知识卡会把愿景、策略、架构、实施路径混在一起，后续难以复用。
- Question: 正式入库时应拆成一张知识卡，还是一张知识卡加一条 ADR？
- Recommendation: 保持当前目标文件计划，但在知识卡中强调适用边界，在 ADR 中强调“不以统一聊天入口为第一阶段目标”。

### 研发交付专家

- Claim: 下一步不应直接 promote，而应先把来源资料审阅和候选审阅走完。
- Evidence: 当前 review queue 仍有 source-note 和 `ds-20260608-002` 等待人工确认；版本状态已能支持回退。
- Risk: 如果跳过来源审阅，正式知识库会沉淀缺证据的战略判断；如果跳过候选审阅，会破坏工作台门禁。
- Question: 是否需要先补真实外部来源或企业内部案例，再 approve 该候选？
- Recommendation: 本 debate 先作为 `source_debates` 补强 `ds-20260608-002`，不生成重复候选。

## Round 2：交叉质询

- 红队质询产品经理：如果业务用户只关心“快点得到答案”，为什么他们会接受指标语义、权限和评测带来的使用成本？
- 产品经理回应：第一阶段必须把治理成本藏在产品体验背后，例如推荐问题、引用来源、可信度提示和一键反馈，而不是要求用户理解底座。
- 架构专家质询战略专家：如果不建设统一 Runtime，如何避免每个业务系统重复造 Agent？
- 战略专家回应：先统一语义、工具注册、权限、Trace 和 Eval，Runtime 可以多实现并存；统一的是治理协议，不一定是单一运行框架。
- 红队质询架构专家：Run Trace 和 Eval 很容易变成日志工程，怎么证明它产生业务价值？
- 架构专家回应：必须绑定黄金问题集通过率、失败回放关闭率、重复问题自动化率和工具调用审计覆盖率。
- 知识编辑专家质询全体：哪些内容现在已经足够稳定，可以进入正式知识库？
- 研发交付专家回应：定位、底座优先级、第一阶段不做统一聊天入口可以作为候选；组织架构、预算、框架选型不能入库。

## Round 3：收敛

- 共识：企业技术部门在 Agent 时代应从项目交付和模型接入，升级为可信智能能力资产的建设和运营方。
- 共识：第一阶段应优先建设可问数据集、指标语义、权限治理、工具边界、运行留痕和评测闭环，而不是统一聊天入口。
- 共识：正式知识必须把“组织定位”“平台能力”“样板场景”“失败判据”拆清楚，避免宏大叙事。
- 分歧：是否需要统一 Agent Runtime 仍未定；当前更稳妥的表述是统一语义、治理、评测协议，运行框架可阶段性多元。
- 分歧：样板场景由哪个团队主导仍需结合组织实际，不能在知识库中写死。
- 最弱假设：技术部门的考核和资源配置能支持能力资产运营，而不是继续以项目需求吞吐为主。
- 最弱假设：业务侧愿意在受控数据集和可审计 Agent 下工作，而不是继续使用个人 AI 工具或影子 IT。
- 待验证：黄金问题集通过率、失败回放关闭率、复用资产数量、工具调用审计覆盖率是否能成为有效验收指标。
- 待验证：一个高频样板场景能否证明资产沉淀比一次性问答有更高复用价值。
