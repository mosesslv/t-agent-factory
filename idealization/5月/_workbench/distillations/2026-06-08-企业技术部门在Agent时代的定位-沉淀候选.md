## 建议入库内容

### 稳定判断

1. 企业技术部门在 Agent 时代的新定位是企业智能能力提供方，而不是传统系统交付团队、报表工厂或模型接入团队。
2. 技术部门的核心建设对象不是聊天入口，而是数据、语义、知识、工具、治理、评测和资产运营底座。
3. 企业技术部门应把能力沉淀为可被人、Agent、BI、业务系统共同消费的资产。
4. 大数据团队的职责应从报表交付升级为 AI-ready 数据资产、语义指标、Agent 工具边界、Run Trace / Eval 和分析资产运营。
5. Agent 能力必须通过运行留痕、评测集、失败回放和反馈闭环持续约束，不能只依赖模型能力。

### 概念定义

1. 企业智能能力提供方：负责把企业数据、指标口径、业务知识、工具能力和治理规则封装成可被人和 Agent 安全消费的组织能力。
2. 能力资产：可复用的数据集、指标语义、分析 Skill、报告素材、工具接口、评测用例、运行反馈和知识卡。
3. Agentic Enterprise Capability Platform：支撑人、Agent、BI 和业务系统共同工作的数据智能能力底座。

### 可复用框架

```text
企业技术部门 Agent 时代职责
= AI-ready 数据资产
+ 指标语义和业务知识
+ Agent 工具与行动边界
+ 受控分析 Skill
+ 分析产物和案例沉淀
+ Run Trace / Eval / AgentOps
+ 集成适配和交付体验
```

### 建设优先级

1. P0：可问数据集、指标语义、权限治理、SQL / Tool Guard、运行留痕、黄金问题集。
2. P1：分析 Skill Hub、报告素材 Asset Hub、知识引用溯源、反馈回流。
3. P2：多 Agent 编排、跨系统行动、自动化报告、企业级 AgentOps。

### 决策记录候选

1. 第一阶段不把目标定义为“建设统一聊天入口”，而定义为“建设可信数据智能底座”。
2. 第一阶段选择企业数据分析作为样板场景，验证从可问数据集到反馈评测的完整闭环。
3. 不把未经验证的组织架构调整、具体 FTE 或框架选型写入正式知识。

## 不建议入库内容

- 未经验证的具体团队组织调整。
- 具体 FTE、预算和排期数字。
- 对 LangGraph、Mastra、OpenAI Agents SDK 等框架的强绑定结论。
- 业务部门是否接受新工作方式的未经验证假设。

## 待验证问题

- 企业技术部门是否具备从项目交付转向能力资产运营的考核和组织条件。
- 业务侧是否愿意通过受控数据集、指标语义和可审计 Agent 做自助分析。
- 黄金问题集、运行留痕和失败回放是否能显著提升问数准确率。
- 统一 Agent Runtime 是否必要，还是先建设统一语义、治理和评测底座更务实。

## 辩论补强

来自 `db-20260609-001` 的红蓝攻防建议：

- 保留：企业技术部门升级为可信智能能力资产建设和运营方。
- 保留：第一阶段优先建设可问数据集、指标语义、权限治理、工具边界、运行留痕和评测闭环。
- 补强：不要把“统一平台”写成唯一方案，更稳妥的表述是“统一语义、治理、评测协议，运行框架可阶段性多元”。
- 补强：正式入库前应增加失败判据，例如无复用、无评测、无反馈、无权限审计时，不得宣称完成 AI 转型。
- 补强：样板场景应有选择标准，优先选择高频、可验证、能形成资产复用的分析闭环。
- 保留边界：组织架构、FTE、预算、具体框架选型仍不进入正式知识库。
