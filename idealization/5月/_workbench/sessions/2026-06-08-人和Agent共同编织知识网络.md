# 研讨 Session：人和 Agent 共同编织知识网络

## 议题

如何把当前 Obsidian/Markdown 知识积累升级为一个人和 Agent 共同工作的知识网络与战略研讨工作台。

## 背景

当前项目已经有 `gather/`、`plan/`、`主题库/` 等知识材料，但仍缺少：

- 人类可读的统一入口。
- Agent 可执行的工作流和边界规则。
- 研讨过程与正式知识之间的隔离。
- 多专家持久化讨论的记录机制。
- 版本管理和回退机制。

用户明确提出，系统不应 100% 依赖 Codex；Codex、Claude Code、ChatGPT、Obsidian 插件或本地脚本都应能接入同一套文件化知识系统。

## 已调取知识

- `主题库/大数据AI智能范式/README.md`: 提供正式主题库结构和 Data Agent 相关核心判断。
- `主题库/大数据AI智能范式/万象链接映射.md`: 提供新主题与已有材料的关联方式。
- `gather/基本共识.md`: 提供 Data Agent 平台化和能力边界的既有共识。
- `gather/能力清单.md`: 提供平台能力树，可作为工作台后续主题材料。

## 人类输入摘要

- 知识库应分为人类展示入口和 Agent 后台入口。
- Obsidian 需要保留面向人阅读的主题入口、索引、图谱和卡片视图。
- 工作台应承载语音或文字输入、多专家持久化研讨、红蓝攻防、沉淀候选和共识形成。
- 不是所有工作台内容都进入知识库，只有被提炼、反复讨论、形成共识的内容才进入正式知识层。
- 系统应可迁移，不应绑定 Codex。
- 第一阶段应轻量可用，不应直接建设大型 Web App 或复杂 Agent runtime。

## 专家观点

### 产品经理

- 核心观点：本次产品目标是建立最小可运行的 Knowledge Workbench V0，而不是直接构建完整 AI 平台。
- 需求判断：P0 是边界、入口、模板、入库规则、专家 profile 和 Git 回退。
- 风险：如果没有工作台与正式知识库隔离，碎片讨论会污染长期知识。

### 战略专家

- 核心观点：该系统本质是人作为议题发起者和最终判断者，Agent 作为研究、辩论、提炼和索引协作者，共同编织知识网络。
- 长期判断：核心壁垒不是某个工具，而是可持续沉淀认知的机制。
- 风险：如果只做聊天体验，会丢失可追溯、可复盘、可演化的知识资产。

### 架构专家

- 核心观点：V0 应使用 Markdown、frontmatter、Obsidian、Git 和模板完成，不引入复杂框架。
- 系统边界：`主题库/` 是正式知识，`_workbench/` 是过程，`_agents/` 是专家视角，`_meta/` 是规则，`_index/` 是索引。
- 风险：过早引入 LangGraph、Mastra 或 Web App 会拖慢最小闭环验证。

### 产品专家

- 核心观点：人类入口必须清晰，用户打开 Obsidian 后应能看到正式知识、最近研讨、待沉淀和专家视角。
- MVP 判断：`00-知识工作台入口.md` 是第一阶段最关键的体验入口。
- 风险：如果入口仍然依赖文件夹浏览，系统不会形成稳定使用习惯。

### 红队专家

- 反对点：多 Agent 不能只是角色扮演，必须有文件化记录、结构化输出和可追溯来源。
- 最弱假设：认为只要有模板，就能保证长期知识质量。
- 需要验证：一次真实议题能否完成“session → distillation → 人确认 → formal knowledge”的闭环。

### 知识编辑专家

- 可沉淀内容：Knowledge Workbench 的边界定义、入库规则、V0 路线、专家角色。
- 不建议入库内容：原始对话、临时工具讨论、未验证的框架选型。
- 需要补证据：后续若引入 Khoj、Mastra、Vercel AI SDK、LangGraph 等，需要基于具体版本和实际试用结果更新技术选型。

### 研发交付专家

- 核心观点：当前项目已建立 Git baseline，后续结构性变更和 promote 都应形成独立提交。
- 实施判断：V0 先不迁移旧文件，只新增结构、规则、模板和入口。
- 回退策略：结构性变更单独提交，promote 单独提交，旧知识通过 deprecated 而不是删除处理。

## 阶段结论

### 共识

- 当前迭代命名为 Knowledge Workbench V0。
- V0 只做 Markdown + Obsidian + Git + 模板。
- 正式知识和工作台过程必须隔离。
- Agent 角色先用 `_agents/*.md` 表达，不先建设真实 Agent runtime。
- Promote 必须人工确认。

### 分歧

- 是否应该在 V1 直接接入 Khoj 或先做简单 CLI，仍待后续试用决定。
- 是否迁移现有 `主题库/` 到更标准的 `kb/topics/` 结构，V0 暂不处理。

### 待验证假设

- Obsidian 入口和 Dataview/Bases 能否满足第一阶段展示需求。
- Markdown profile 是否足够支撑多专家持久化讨论。
- Git 提交粒度能否满足知识演进和回退。
