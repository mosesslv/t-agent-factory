# 研讨 Session：企业技术部门在 Agent 时代的定位

## 议题

Agent / LLM 冲击下，企业技术部门未来应提供什么、建设什么，以及如何避免停留在“接入模型和聊天入口”的浅层改造。

## 背景

已有知识库已经形成几个关键判断：

- 企业数据智能不是单一 Chatbot，而是数据、语义、知识、工具、治理、评测和资产沉淀组成的能力体系。
- 大数据团队不再只是报表交付团队，而是企业智能分析能力提供方。
- Data Agent 平台需要数据资产、可问数据集、语义指标、知识中心、工具执行、Skill Hub、Asset Hub、治理和 Run Trace / Eval。
- DB-GPT 规划材料已经指出 NL-to-SQL、数据配置、解读、洞察、报告、语义层和多阶段分析编排是关键短板。

## 人类输入

用户关注的不是一次性答案，而是企业技术部门如何在 Agent / LLM 冲击下重新定义自身职责、能力建设和工作方式。该议题需要经过多轮研讨、质疑、推翻和沉淀，最终形成可复用的战略判断。

## 专家观点

### 产品经理

- 核心观点：企业技术部门不能把 Agent 时代的需求理解成“上线一个内部 ChatGPT”或“给系统加自然语言入口”。
- 需求判断：业务真正需要的是可信、可用、可治理、可复用的智能能力底座。
- P0 能力：数据资产可发现、指标语义可复用、知识上下文可绑定、工具调用可审计、运行效果可评测。
- 风险：如果只做前端聊天入口，短期演示效果好，但无法支撑企业级准确率、权限、追溯和持续运营。

### 战略专家

- 核心观点：企业技术部门的新定位是“组织智能能力提供方”，不是传统 IT 支撑、数据报表工厂或模型接入团队。
- 长期判断：未来技术部门的核心价值从交付系统功能，转向沉淀企业可被人和 Agent 共同消费的能力资产。
- 组织判断：业务侧负责问题、判断和行动；技术侧负责可信数据、语义、工具、治理、评测和资产运营；Agent 负责检索、查询、归因、草拟、编排和执行辅助。
- 风险：技术部门如果不掌握语义、权限、评测和工具治理，会被业务侧碎片化 AI 工具绕开，形成新的影子 IT。

### 架构专家

- 核心观点：企业技术部门应建设“Agentic Enterprise Capability Platform”，而不是分散建设多个聊天应用。
- 架构边界：平台层沉淀通用能力，应用层承载不同工作流。
- 平台能力包括：Data Center、Dataset Learning Center、Semantic & Metric Center、Knowledge Center、Agent Runtime、Tool & Execution Hub、Skill Hub、Asset Hub、Governance & Security、Run Trace & Eval、Integration Adapter。
- 风险：没有运行留痕和评测闭环，Agent 能力会变成不可控黑盒；没有统一工具注册和权限边界，Agent 行动会带来安全风险。

### 产品专家

- 核心观点：第一阶段产品不应追求“覆盖所有业务”，而应选择一个能展示完整闭环的样板场景。
- MVP 建议：以企业数据分析为样板，打通“可问数据集 -> 语义指标 -> 问数/解读 -> 报告素材 -> 反馈评测 -> 资产沉淀”。
- 用户体验：业务用户不需要理解底层架构，但需要看到数据集说明、推荐问题、指标口径、引用来源、结果可信度和可复用素材。
- 风险：如果 MVP 不含反馈和沉淀，系统仍然只是一次性问答工具。

### 红队专家

- 反对点：这些判断容易变成宏大叙事，落地时仍然回到“买模型、接 API、做 Demo”。
- 最弱假设：企业技术部门有能力从项目交付转向资产运营，但组织考核和资源配置未必支持这种变化。
- 必须补强：需要把新职责拆成可交付资产、流程和指标，例如可问数据集数量、指标语义覆盖率、黄金问题集通过率、Agent 工具调用审计率、失败回放闭环率。
- 是否阻止入库：不阻止，但正式知识必须标注为战略判断和阶段性框架，不能写成已验证事实。

### 知识编辑专家

- 可沉淀内容：企业技术部门新定位、六类核心职责、建设优先级、MVP 样板闭环、风险和评测指标。
- 不建议入库内容：未经验证的具体组织架构调整、具体 FTE 数量、未评估的框架选型。
- 需要补证据：后续应补充企业内真实项目约束、组织职责边界、已有平台能力现状和业务侧高频场景。

### 研发交付专家

- 实施判断：本主题应先形成战略知识卡和决策记录，再进入具体项目计划。
- 交付路径：V0 先沉淀主题框架；V1 选一个样板场景；V2 建立评测和资产运营指标；V3 再考虑真实多 Agent 编排。
- 回退策略：如果后续实践证明“统一平台”过重，可以降级为“统一语义/治理/评测底座 + 多应用适配”。

## 阶段结论

### 共识

1. 企业技术部门在 Agent 时代的定位应升级为企业智能能力提供方。
2. 核心建设对象不是聊天入口，而是数据、语义、知识、工具、治理、评测和资产运营底座。
3. 技术部门应把能力沉淀为可被人、Agent、BI、业务系统共同消费的资产。
4. MVP 应选择能展示完整闭环的场景，而不是铺开所有业务。
5. 需要用运行留痕、评测和反馈闭环约束 Agent 能力质量。

### 分歧

1. 第一阶段是否应由大数据团队主导，还是由平台工程/AI 平台/业务数字化团队共同主导，仍需结合组织实际判断。
2. 是否建设统一 Agent Runtime，还是先建设统一语义、治理和评测底座，仍需结合现有技术栈判断。
3. ChatBI、DB-GPT、QuickBI 小Q 等既有系统之间的边界仍需要单独架构评估。

### 待验证假设

1. 企业技术部门是否具备从项目交付转向能力资产运营的组织条件。
2. 业务侧是否愿意通过受控数据集、指标语义和可审计 Agent 进行自助分析。
3. 样板场景能否证明“资产沉淀”比“一次性问答”有更高复用价值。
4. 黄金问题集、运行留痕和失败回放是否能显著提升 Agent 问数准确率。
