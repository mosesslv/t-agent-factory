## 2. 一句话摘要

传音的 Data Agent 建设，不是简单给 BI 加一个聊天框，而是把数据连接、黄金数据集、指标口径、业务知识、分析技能、运行审计和报告产物组织成一套企业级智能分析能力，让业务用户从“看报表、提需求、等取数”逐步走向“可问数、可解释、可复用、可运营”的智能分析模式。

## 3. 核心观点

1. Data Agent 的核心不是模型，而是 **AI-ready 的数据资产和可信运行体系**。
2. 企业数据分析不会被单一聊天入口替代，未来会是 ChatBI、传统 BI、智能入口、报告系统和 Data Agent 底座协同。
3. DB-GPT 这类开源框架适合作为 Agent Runtime 和多源分析实验底座，但不能直接等同于成熟 BI 或 QuickBI 小Q。
4. 企业级落地的关键补齐项是：可问数据集、指标/知识注入、SQL 安全、分析编排、报告资产化、反馈评测闭环。
5. 建设路径应从小闭环开始：个人文件问数、DB 可信问数、经营报告生成，再逐步扩展到看板解读、洞察归因和行动型 Agent。

## 3.1 可补充的企业背景框架

从历史文档看，这次分享可以把“传音背景”抽象为一个大型企业数据分析平台的典型演进问题：底层数仓、宽表、指标、BI 看板已经积累了一批资产，但业务侧的分析需求正在从“看固定报表”走向“自然语言提问、解释变化、诊断原因、生成经营材料”。因此，Data Agent 的建设不是单点替代 BI，而是把已有数据资产升级成 AI 可以理解、可以调用、可以运营的分析服务。

建议对外表达时避免过度展开内部系统细节，可以讲成三个背景特征：

| 背景特征 | 对 Data Agent 的要求 |
|---|---|
| 多角色使用 | 管理层看经营结论，业务人员问数追因，分析师沉淀模板，数据团队治理口径 |
| 多数据对象并存 | 个人 Excel、主题宽表、语义数据集、指标、看板、报告、业务知识都要被统一理解 |
| 多分析深度并存 | 从查数、探索、解释、归因，到洞察、报告、行动协同，复杂度逐层上升 |

这能支撑一个关键判断：

> 企业 Data Agent 不是“问一句、答一句”的聊天机器人，而是把数据对象、分析动作和交付产物统一编排起来的企业分析基础设施。

## 3.2 企业分析复杂度：数据对象 x 分析自由度 x 治理等级

历史文档里有一个很适合外部分享的抽象：企业数据分析不能只按“自然语言问答”理解，而要同时看三件事：

```text
数据对象粒度 x 分析自由度 x 可信治理等级
```

数据对象粒度决定系统理解什么数据：

| 粒度 | 数据对象 | 典型意义 |
|---|---|---|
| D0 | 个人 Excel / CSV / 临时文件 | 低治理、快速探索、适合个人问数 |
| D1 | 单表 / 领域宽表 | 已建模但语义有限，适合主题域查数 |
| D2 | 数据集 / 语义数据集 | 定义字段、关系、维度、口径，是问数核心对象 |
| D3 | 指标 / 指标体系 | 口径稳定、可复用、可治理，是管理决策对象 |
| D4 | 报表 / 看板 / 仪表板 | 已完成可视化组织，适合解读、追问和洞察 |
| D5 | 报告 / 经营分析材料 | 多章节叙事、图文结合，需要审核和版本 |
| D6 | 业务知识 / 方法论 / SOP | 解释业务术语、诊断框架和异常判断规则 |

分析自由度决定用户任务复杂度：

| 自由度 | 用户任务 | 示例 |
|---|---|---|
| F0 | 查询事实 | “上月销售额是多少？” |
| F1 | 切片探索 | “按区域看 TOP10” |
| F2 | 指标解释 | “为什么毛利率下降？” |
| F3 | 多步诊断/归因 | “哪个事业部拖累利润？” |
| F4 | 经营洞察 | “下月应重点关注什么风险？” |
| F5 | 报告生成 | “生成本月经营分析报告” |
| F6 | 行动建议/Agent 执行 | “把异常生成任务并通知负责人” |

可信治理等级决定能不能用于经营决策：

| 等级 | 含义 | 适用场景 |
|---|---|---|
| G0 临时分析 | 用户自担口径，系统提示字段和质量风险 | 个人 Excel、临时 CSV |
| G1 可复用分析 | 数据对象可保存复用，有基础字段描述 | 宽表、单表数据集 |
| G2 标准问数 | 字段、指标、维度、口径可治理 | 企业数据集、主题域 |
| G3 管理决策 | 指标经业务确认，结果可审计 | 指标体系、看板、经营报告 |
| G4 自动行动 | 可触发任务/通知/流程，需要权限审批 | 告警、订阅、Agent 执行 |

这部分适合作为分享中的“方法论页”：它解释了为什么企业 Data Agent 必须建设数据集、指标、知识、权限、运行态和评测，而不是只做一个模型接口。

## 3.3 传音场景下的产品分工建议

历史文档里的产品分工可以转成一页很清晰的对外表达：

| 产品/能力 | 更适合承担的职责 | 不建议承担的职责 |
|---|---|---|
| FineBI / 传统 BI | 固定报表、经营看板、复杂可视化编辑和门户分发 | 深度推理、自动归因、长文本经营报告生成 |
| ChatBI / 小Q式入口 | 面向业务用户的自然语言问数、多轮追问、结果展示 | 独自承担所有底层数据治理和复杂工作流编排 |
| DB-GPT / Data Agent Runtime | 深度分析、报告生成、Skill 编排、私有化和模型自由度 | 替代完整 BI、替代数仓建模、绕过指标口径 |
| 数仓/指标/知识平台 | 数据正确性、指标口径、业务术语、诊断框架 | 直接承担所有前台交互体验 |

一句话版：

> FineBI 负责稳定展示，ChatBI 负责自然语言入口，Data Agent 负责把问数、解读、洞察、报告编排成可追踪的企业分析服务。

## 4. 建议分享大纲

### 4.1 开场：为什么企业数据分析需要 Data Agent

可讲问题：

- 业务分析需求越来越碎片化，传统“提需求-取数-做报表”链路响应慢。
- BI 已经沉淀了大量报表和看板，但业务仍然需要解释、追问、归因和报告。
- 大模型让自然语言交互成为可能，但直接让模型猜表、猜字段、猜口径并不可靠。
- 企业需要的是可控、可审计、可复用的智能分析能力，而不是单点 Demo。

可用表达：

> Data Agent 的价值不是替代数据团队，而是把数据团队沉淀的数据资产、指标口径和分析方法，以更低门槛交给业务用户使用。

### 4.2 第一部分：从“报表交付”到“分析资产交付”

传统数据部门的交付方式：

```text
业务提需求 -> 数据部门取数/做报表 -> 业务看报表 -> 再提新需求
```

面向 Data Agent 的目标模式：

```text
大数据部门沉淀 AI-ready 数据资产和分析能力
-> 业务用户通过问数、解读、报告自助消费
-> 高频问题和优质答案沉淀回模板、知识库和评测集
```

大数据部门未来需要提供的不是单一工具，而是一组分层资产：

| 层级 | 资产 | 说明 |
|---|---|---|
| L0 | 数据连接与原始数据 | 数据源接入、权限账号、连接管理，给智能分析安全访问数据 |
| L1 | 标准表与领域宽表 | 主题宽表、标准字段、稳定刷新，是可问数的数据底座 |
| L2 | 可问数据集 | 字段含义、别名、样例问题、可问范围和质量画像 |
| L3 | 指标与业务知识 | 指标口径、业务术语、诊断框架、组织/区域/产品知识 |
| L4 | 分析技能 | 问数、解读、归因、报告等可复用 Skill |
| L5 | 分析产物 | 问数结果、洞察卡、报告素材、引用关系 |
| L6 | 运营反馈 | 准确率、成功率、成本、延迟、用户反馈、评测集 |

这一部分的 Key Message：

> BI 时代的核心资产是表、指标和报表；AI 时代还要补齐可问数据集、分析技能、报告素材和反馈评测。

### 4.3 第二部分：Data Agent 在企业数据体系中的定位

Data Agent 的定位是中间能力层：

```text
业务入口层：T-Claw / PI-Claw / ChatBI / 小Q式入口 / FineBI / 报告生成入口
        ↓
Data Agent：统一编排、语义注入、安全运行、产物沉淀、评测运营
        ↓
数据资产层：文件、库表、主题宽表、指标口径、业务知识、BI 看板
```

Data Agent 不做什么：

- 不重做数仓，不替代数据建模和数据质量治理。
- 不重做传统 BI，不替代复杂报表和门户。
- 不替代业务入口，T-Claw、ChatBI、小Q式入口、BI 门户都可以调用同一套底座能力。
- 不绕过权限、安全、审计和数据源口径。

Data Agent 重点做什么：

| 能力 | 说明 |
|---|---|
| 接得上 | 统一接入数据、指标、知识、图表和文件 |
| 问得准 | 把字段别名、指标口径、业务术语和样例问题带入问数过程 |
| 跑得稳 | 编排问数、解读、洞察、报告等多步骤任务 |
| 管得住 | 控制只读、限量、超时、敏感字段、权限和审计 |
| 留得下 | 沉淀结果、图表、洞察、报告、引用关系和人工反馈 |
| 评得清 | 持续观察准确率、成功率、响应时间、成本和用户反馈 |

这一部分的 Key Message：

> Data Agent 是连接数据资产与业务入口的中间平台，让分析能力可用、可信、可复用。

### 4.3.1 可讲的场景地图：16 类企业智能分析场景

为了让外部观众更直观看到建设范围，可以把场景压缩成 4 组、16 类：

| 场景组 | 代表场景 | 建设含义 |
|---|---|---|
| 个人文件分析 | 个人文件模板化报表、字段探索、自然语言自由提问、数据质量洞察 | 覆盖业务人员临时 Excel/CSV 分析，快速建立智能分析体验 |
| 领域数据集分析 | KPI 仪表盘、多维分析、NL-to-SQL 自由探索、异常自动发现 | 从“查表”升级到“问业务数据集”，是企业问数主战场 |
| 指标分析 | 指标看板、指标归因、指标自然语言对话、指标预警 | 面向经营管理口径，支撑同比环比、目标达成和责任归因 |
| 报告分析 | 周期性报表、报告定制、自然语言生成报告、智能周报/月报 | 把数据结果转成管理层可消费、可审核、可复用的经营材料 |

当前材料中对 DB-GPT 的判断是：

- 高参与度核心场景：个人文件 NL 提问、数据质量洞察、领域数据集 NL-to-SQL、异常发现、指标归因、指标对话、指标预警、报告定制、NL 生成报告、智能周报/月报。
- 协同场景：个人文件字段探索、领域数据集多维分析、周期性报表。
- 低参与或非主战场：个人文件固定模板报表、固定 KPI 仪表盘、传统指标看板。

对外表达可以更简洁：

> 固定展示和成熟报表仍交给 BI；Data Agent 重点进入高自由度、高解释需求、高叙事需求的场景。

### 4.4 第三部分：我们已经探索过什么

当前材料中可整理出的探索方向如下，真实上线状态待补充。

#### 方向 1：个人文件智能分析

面向 Excel/CSV 上传后的即问即答和轻量探索。

已探索/可探索能力：

- 文件上传后自动识别 Sheet、字段、类型和样例值。
- 基于 DuckDB 或类似引擎做本地表格查询。
- 用户用自然语言提问，系统返回表格、图表和基础摘要。
- 文件分析结果沉淀为个人素材，后续可升级为部门数据集或报告素材。

价值：

- 启动成本低，适合快速验证用户体验。
- 能覆盖临时分析、个人探索、轻量数据检查。

局限：

- 数据口径和质量主要由用户自担。
- 不适合直接作为企业管理决策来源。

#### 方向 2：DB/数仓可信问数

面向主题宽表、领域数据集和数仓表的自然语言问数。

已探索/可探索能力：

- 连接数据库和数仓表。
- 根据自然语言生成 SQL，并返回查询结果。
- 展示 SQL 过程，支持预览、编辑和错误修复。
- 引入 DatasetService、字段别名、样例问题、指标口径和可问范围。
- 引入 SQL 安全网关，控制只读、限行、超时、敏感字段和审计。

价值：

- 从“业务问人取数”转向“业务自助问数”。
- 通过可问数据集和 SQL 过程追溯提升可信度。

关键难点：

- NL-to-SQL 准确率。
- 业务口径和字段语义。
- 权限、安全、审计。
- 错误样例和评测集运营。

#### 方向 3：看板/图表智能解读

面向已有 BI 看板、图表和经营监控结果的自动解释。

已探索/可探索能力：

- 对接 FineBI/QuickBI/自有 dashboard 的图表数据和筛选上下文。
- 读取图表标题、指标、维度、筛选条件、时间范围和底层数据。
- 生成趋势、Top/Bottom、异常点、同比/环比和结构变化解读。
- 按“现象 -> 原因 -> 影响 -> 建议”输出复盘文本。

价值：

- 不改变用户已有看板使用习惯。
- 从“看见数据”走向“理解数据”。

关键难点：

- 不能只把截图丢给模型解释，必须把图表元数据、底层数据和业务口径一起注入。
- 多图之间的链路、因果假设和互相验证需要结构化工作流。

#### 方向 4：经营报告生成

面向周报、月报、专题复盘和管理汇报。

已探索/可探索能力：

- 通过模板定义报告章节、数据依赖和输出风格。
- 把问数结果、图表和洞察卡进入报告素材池。
- 生成 Markdown/HTML/PPT/PDF 等可交付形态。
- 保存 ReportRun、Artifact、Trace 和引用关系。
- 支持人工编辑、审批、导出和订阅。

价值：

- 把分析结果转成管理层可消费材料。
- 是 DB-GPT/Data Agent 相对传统 BI 的差异化展示方向。

关键难点：

- 报告不能只有“自然语言总结”，必须有证据链。
- 每段结论应关联数据、SQL、图表、指标口径和时间范围。

#### 方向 5：分析 Skill 与模板市场

面向不同业务域沉淀可复用分析方法。

Skill 不是一个 prompt，而是一组完整能力包：

- 能力说明：适合解决什么问题，不适合什么问题。
- 输入契约：需要数据集、图表、文件、指标、时间范围还是自然语言问题。
- 工具链：NL-to-SQL、OLAP 查询、图表生成、解读、报告、外部系统 API。
- Prompt/模板：意图识别、SQL 生成、解释、报告章节、错误修复。
- 安全边界：可访问哪些数据、可执行哪些 SQL、是否需要审批。
- 输出契约：表格、图表、洞察卡、报告、任务、链接。
- 评测样例：典型问题、标准 SQL、标准答案、失败案例。

这一部分的 Key Message：

> 真正可复制的不是某一次问答，而是被沉淀下来的数据集、指标、Skill、模板、报告素材和评测样例。

### 4.5 第四部分：DB-GPT 在建设中的角色

当前判断：

**DB-GPT 可以作为企业级数据分析平台的 Agent 执行底座、报告生成底座和多源分析实验底座，但不能直接等同于 QuickBI 小Q。**

DB-GPT 适合复用的能力：

| 能力 | 说明 |
|---|---|
| 多数据源连接 | MySQL、PostgreSQL、Spark、DuckDB、ClickHouse、Doris、StarRocks 等 |
| Chat Scene | ChatDB、ChatData、ChatExcel、ChatDashboard 等场景原型 |
| LLM 调用 | 统一模型调用和 Prompt 执行基础 |
| SQL 执行 | 可作为 NL-to-SQL 和数据库对话原型 |
| Excel/CSV 分析 | 适合个人级文件分析 Demo |
| Dashboard/报告原型 | 可作为图表生成和报告生成探索起点 |
| 私有化和开放可控 | 适合企业内部二开和模型自由度要求 |

DB-GPT 当前不足：

| 缺口 | 影响 |
|---|---|
| 无成熟数据集配置层 | 模型直接理解表结构，缺少业务语义 |
| SQL 安全不足 | 裸 SQL 执行存在权限、限行、超时和审计风险 |
| 解读能力弱 | 查询结果缺少结构化解释、异常识别和归因 |
| 报告缺少资产化 | 需要 ReportRun、Artifact、Trace、引用关系 |
| 缺少运营评测 | 无法持续跟踪准确率、成功率、成本和失败样例 |
| Scene 架构偏单点 | 需要从 Scene Router 演进为 Capability Orchestrator |

建议定位：

```text
DB-GPT 不做完整 BI 产品
DB-GPT 不直接复制小Q
DB-GPT 做开放可控的 Data Agent Runtime
```

### 4.6 第五部分：目标架构和关键模块

目标架构可以抽象为：

```text
用户接入层
Web UI / REST API / WebSocket / ChatBI / T-Claw / BI Portal

Capability Orchestrator
问数引擎 / 解读引擎 / 洞察引擎 / 报告编排

AnalysisChain
意图识别 -> 语义注入 -> SQL 生成 -> 安全校验 -> 执行 -> 解读 -> 洞察 -> 报告

基础设施层
LLM 路由 / 数据配置层 / 向量检索 / 缓存 / 连接池 / Prompt 注册 / 多租户隔离

资产与治理层
Dataset / Metric / Knowledge / Skill / Template / Run / Trace / Artifact / Feedback / Eval
```

核心平台对象：

| 对象 | 作用 |
|---|---|
| DataSource | 数据源连接、认证和权限边界 |
| DataSet | 可问数据对象，承载字段、主题、可问范围和质量画像 |
| DataField | 字段名、别名、描述、类型、单位、维度/度量属性 |
| MetricCatalog | 指标口径、公式、维度层级、版本 |
| KnowledgeBase | 业务术语、SOP、诊断框架、历史报告 |
| Skill Registry | 问数、解读、归因、报告等能力注册和路由 |
| AskRun | 一次问数任务的状态、SQL、结果、错误和审计 |
| InsightRun | 一次解读/洞察任务的输入、算法、结论和证据 |
| ReportRun | 一次报告生成任务的章节、素材、输出和版本 |
| Artifact | 表格、图表、洞察卡、报告片段、引用关系 |
| Feedback/Eval | 用户反馈、评测样例、准确率和回归测试 |

### 4.7 第六部分：阶段性建设路线

#### 阶段 0：底座验证与技术拆解

目标：证明 DB-GPT 或 Data Agent Runtime 能作为统一分析底座。

交付：

- 可运行环境和模型配置。
- 数据源连接验证。
- Excel/CSV 文件问数验证。
- DB 问数链路验证。
- 报告生成 Demo 验证。
- 技术边界确认：DB-GPT 做 Agent，ChatBI 做可信指标问数，BI 做 dashboard 数据来源。

#### 阶段 1：4-6 周可演示闭环

目标：形成外部/管理层能感知的 Demo。

建议能力：

- Excel/CSV 上传问数，返回表格、图表和基础摘要。
- DB 问数，生成 SQL，支持预览、编辑和安全拦截。
- fin-report JSON 输入生成 4 章经营报告，前端可查看。
- ChatBI 能力以 Skill 方式接入，复用已有可信问数。

参考投入：

- MVP 约 64-95 人天。
- 3-5 人小队，4-6 周形成可演示闭环。

可对外讲的验收口径：

| 验收项 | 建议目标 |
|---|---|
| Excel/CSV 问数 | 上传后可基于自然语言返回表格、图表和基础摘要 |
| DB 问数 | 支持 SQL 预览、编辑、安全拦截和错误修复 |
| 报告生成 | 支持 3-4 个章节的 Markdown/HTML 经营报告 |
| 过程追溯 | 至少保存问题、SQL、结果、报告产物和错误信息 |
| 演示稳定性 | 选定演示数据和问题集，保证端到端可重复 |

#### 阶段 2：10-16 周平台雏形

目标：补齐可问数据集、解读和报告主线。

建议能力：

- DatasetService：字段别名、字段描述、样例问题、可问范围。
- SqlSafetyGateway：只读、限行、超时、敏感字段、审计。
- AskRun/ReportRun/Artifact：让过程和产物可追踪。
- 看板一键解读：图表数据 + 元数据 + 指标口径。
- 统一 Skill 入口：问数、图表、洞察可进入报告素材链。

参考投入：

- 平台雏形约 182-283 人天。
- 5-7 人小队，10-16 周形成数据集、解读、报告编辑主线。

历史规划中的产品化指标可以作为内部目标，不一定全部对外披露：

| 指标 | 参考目标 |
|---|---|
| NL-to-SQL 准确率 | MVP 阶段目标 >80%，成熟数据集上进一步提升 |
| 查询结果解读覆盖率 | 超过 60% 的查询附带结构化解读 |
| 报告生成成功率 | 端到端报告生成成功率 >90% |
| 报告生成时间 | 4 章节报告小于 5 分钟 |
| 反馈闭环 | 支持用户纠错、收藏、采纳、差评和失败样例沉淀 |

#### 阶段 3：企业增强

目标：从 Demo 和试点变成企业级服务。

建议能力：

- 指标目录和口径版本管理。
- 归因洞察和专题分析模板。
- 多租户、权限、审计和成本治理。
- 评测集、反馈闭环和运营看板。
- 报告审批、导出、订阅和分发。
- 行动型 Agent：告警、任务生成、通知负责人，但应在权限和审计成熟后推进。

参考投入：

- 企业增强约 365-585 人天。
- 重点补指标目录、归因洞察、真实业务模型、审计权限和导出。

### 4.7.1 工程风险和缓解策略

这部分适合放在技术分享后半段，体现实践深度。

| 风险 | 为什么重要 | 缓解方向 |
|---|---|---|
| NL-to-SQL 准确率不达标 | 错 SQL 会直接误导业务判断 | 数据集语义层、Few-shot、SQL 校验、错误修复、评测集 |
| 多阶段调用延迟高 | 问数、解读、洞察、报告会串联多个模型和工具调用 | Stage 并行、缓存、流式输出、模型路由 |
| LLM 输出不稳定 | 影响 SQL、图表、报告结构解析 | 结构化输出、重试、降级、模板约束 |
| 数据安全和权限 | 企业分析系统不能绕过原有权限体系 | 行列权限、只读 SQL、敏感字段脱敏、审计日志 |
| 报告质量不可控 | 纯 LLM 报告容易空泛或虚构原因 | 模板驱动、证据链、引用数据、人工审核 |
| 成本失控 | 高频问数和报告生成会带来模型和查询成本 | Token 配额、结果缓存、小模型处理简单任务 |

可讲经验：

> 企业级智能分析系统要把“正确性、安全性、可解释性、成本”放在和“生成效果”同等重要的位置。

### 4.8 第七部分：实践经验和踩坑

可对外分享的经验点：

1. 不要先追求“万能问答”，先定义可问范围。
2. 不要只优化 Prompt，要先补字段、指标、样例问题和业务术语。
3. 不要让模型直接执行不受控 SQL，要有 SQL 安全网关。
4. 不要把报告生成做成纯文本总结，要保存证据链和素材引用。
5. 不要把 Data Agent 当作 BI 替代品，要让它和已有 BI、ChatBI、数仓、指标平台协同。
6. 不要上线后就结束，要持续运营准确率、失败样例、用户反馈和成本。

可用表达：

> 企业 Data Agent 的难点不是让模型回答一次，而是让每一次回答都能被解释、被追溯、被改进。

### 4.9 结尾：未来展望

短期：

- 从个人文件和 DB 问数切入，形成用户能体验的智能分析闭环。
- 把已有 ChatBI/FineBI/数仓能力接入 Data Agent，而不是重复建设。
- 先做可信问数、报告生成和看板解读三个高感知场景。

中期：

- 建设黄金数据集、指标口径、业务知识库和 Skill Hub。
- 把问数结果、洞察卡、图表和报告片段沉淀为分析素材。
- 建立反馈、评测、成本和质量运营体系。

长期：

- 从“问数型 Agent”演进到“分析型 Agent”。
- 从“分析结果生成”演进到“经营任务协同”：异常发现、归因、生成任务、通知负责人、跟踪闭环。
- 让大数据部门从报表交付团队升级为企业智能分析能力提供方。

结尾句候选：

> Data Agent 不是一个新的聊天窗口，而是一套把企业数据资产转化为智能分析服务的能力体系。它的长期价值，在于让数据、知识、分析方法和业务行动形成持续循环。
