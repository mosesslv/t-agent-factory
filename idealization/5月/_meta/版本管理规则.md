# 版本管理规则

## 基线

当前项目已初始化 Git，基线提交为：

```text
8fe6be9 chore: establish knowledge base baseline
```

这个提交代表引入 Knowledge Workbench V0 前的项目状态。

## 提交策略

- 结构性变更单独提交。
- 每次 promote 到正式知识库单独提交。
- 提交信息应包含动作、主题或 session id。
- 不把多个无关主题的知识更新混入同一提交。

建议提交信息：

```text
feat: add knowledge workbench v0 structure
docs: add session ws-20260608-001
docs: promote knowledge card kc-20260608-001
docs: deprecate outdated claim in topic xxx
```

## 回退策略

- 小范围错误优先使用 `git revert`。
- 单文件错误可从历史提交恢复该文件。
- 大规模结构迁移前先创建压缩备份。
- 不直接删除正式知识，除非用户明确要求。
- 正式 promote 前先运行 `tools/kb promote-audit <distillation-file>`，确认计划写入文件、Git 状态、提交边界和回退方式。

## 知识演进策略

- 旧结论被替代时，优先标记 `deprecated`。
- 新结论应记录 `supersedes` 或 `related_claims`。
- 正式知识卡应记录 `source_sessions`。
- 决策记录应保留取舍原因，而不是只保留结论。

## Git 注意事项

当前 `.gitignore` 排除了：

- `.DS_Store`
- Obsidian workspace 本地状态文件
- `_index/generated-version-status.md/json` 本地版本状态快照
- `_index/generated-promote-audit.md/json` 本地 promote 审计快照
- `_index/generated-agent-brief.md/json` 本地 Agent 接手快照

如果后续引入缓存、索引数据库或 Web App 构建产物，应继续扩展 `.gitignore`，避免把运行时噪音提交到知识库。

## 只读状态入口

使用下面命令查看当前版本状态：

```bash
tools/kb version-status
tools/kb version-status --json
```

该命令只读取 Git 状态，不执行 `revert`、`checkout`、`reset` 或分支切换。

`tools/kb refresh-index` 会生成 `_index/generated-version-status.md/json` 供 Obsidian 和 Agent 读取。该快照反映本地运行态，不提交进 Git。

`tools/kb refresh-index` 也会生成 `_index/generated-promote-audit.md/json`。该审计包包含当前 Git HEAD 和 dirty 状态，属于本地运行态，不提交进 Git；它不代表已经执行 promote。

`tools/kb refresh-index` 还会生成 `_index/generated-agent-brief.md/json`。该接手包包含当前 Git HEAD、dirty 状态、最高优先级审阅任务和安全规则，属于本地运行态，不提交进 Git；它用于让 Codex 之外的客户端快速接手当前工作台。

`tools/kb refresh-index` 会生成 `_index/generated-tool-catalog.md/json`。该目录不包含本地 Git 状态，是可提交的工具契约快照，用于 MCP server、Obsidian 插件或其他 Agent 客户端识别工具权限和风险。
