## 1. 执行摘要

Snowflake Summit 26 释放的信号非常清晰：Snowflake 正在从“云数据仓库 / 数据云”进一步转向“企业 AI Agent 的数据、语义、治理、安全与执行底座”。它不只是发布了几个 AI 功能，而是在重新定义企业 AI 的基础架构：

```text
数据不搬走
模型进入数据平台
语义成为 Agent 可信上下文
治理从数据访问扩展到 Agent 行动
业务用户和开发者都通过 Agent 消费数据
```

这个方向与我们当前围绕 DB-GPT、ChatBI、ChatExcel、ChatReport、Quick BI 小Q形成的判断一致：未来企业数据智能不是单一聊天入口，而是一个由数据资产、语义口径、业务知识、工具执行、权限治理、运行评测和分析资产沉淀组成的能力体系。

因此，大数据部门未来要提供的不是“更多报表”或“一个问数机器人”，而是一套可被业务用户、BI 工具、个人 Agent、多 Agent 工作流共同调用的企业智能分析底座。

## 2. Snowflake Summit 26 的关键战略信号

### 2.1 AI 要进入数据平台，而不是把数据搬去 AI 平台

Snowflake 与 Anthropic 的官方发布强调 Claude 通过 Cortex AI 直接运行在 Snowflake 的数据、治理、安全和协作环境中，客户可以在不把敏感数据移出 Snowflake 的情况下部署 AI agents。Reuters 对 Snowflake 与 OpenAI 合作的报道也指向同一逻辑：把 OpenAI 模型嵌入 Snowflake 数据平台，让企业在保留安全与治理控制的前提下分析数据、搜索内部文档、自动化工作流。

这说明企业 AI 的基础设施竞争正在从“谁有更强模型”扩展到“谁能让模型安全、准确、持续地作用于企业数据”。

### 2.2 Agent 成为数据平台的新用户

过去数据平台的主要用户是人：数据工程师写 pipeline，分析师写 SQL，管理者看 dashboard。Summit 26 的产品叙事变成了：Agent 会查询数据、理解语义、生成 pipeline、创建 dashboard、调用外部系统、执行 workflow。

Snowflake CoWork 面向知识工作者，强调从洞察到行动；Snowflake CoCo 面向 builder，帮助开发者、数据工程师、分析师自动化工作流和构建应用。两者背后的共同点不是“聊天”，而是把数据平台推向“行动系统”。

### 2.3 语义层从 BI 辅助层变成 Agent 可信上下文层

Horizon Context / Horizon Catalog 是 Summit 26 中最关键的底座信号。Snowflake 强调每个人、工具和 AI agent 都需要基于同一套可信业务上下文工作。原因很直接：当 Agent 开始自主决策，指标定义、字段含义、业务规则的小偏差会被放大成严重业务错误。

这与我们已有判断一致：Data Agent 的准确率问题不能靠裸 Text2SQL 解决，必须靠数据集学习、指标语义、业务知识和运行反馈共同约束。

### 2.4 治理从“数据访问控制”升级为“Agent 行为控制”

Natoma 的 MCP 治理能力被 Snowflake 纳入后，治理对象从“谁能看什么数据”扩展为“Agent 能连接什么工具、以什么身份调用、能执行什么动作、如何审计”。这对企业 Data Agent 建设非常重要。

未来的数据治理对象至少包括：

- 数据访问：表、字段、行权限、脱敏、分类分级。
- 语义使用：指标口径、默认过滤、业务定义、版本。
- 工具调用：SQL、Python、BI、飞书、CRM、工单、API。
- 行动权限：发送消息、创建任务、修改配置、触发流程。
- 运行证据：plan、tool call、SQL、输入输出、耗时、成本、反馈。

### 2.5 开放湖仓与互操作成为 AI 控制面的竞争基础

Snowflake 的开放互操作框架、Apache Iceberg v3、Horizon Catalog powered by Apache Polaris、Zero-Copy Integrations 等发布，说明它不想只管理 Snowflake 内部数据，而是想成为跨平台数据和 AI Agent 的治理控制面。

这对企业内部大数据平台也有启发：未来不应把 Data Agent 建成某个数据库、某个 BI、某个聊天工具的附属功能，而要围绕“统一上下文、统一权限、统一评测、统一资产”建设跨工具底座。

## 3. 大数据技术演进主线

可以把近十多年大数据技术演进整理成五个阶段：

| 阶段 | 核心技术形态 | 核心交付 | 主要用户 | 局限 |
|---|---|---|---|---|
| 数据仓库阶段 | MPP、数仓建模、ETL | 标准表、主题域、报表 | 数据团队、BI 用户 | 交互慢，需求排队 |
| 湖仓阶段 | Lakehouse、Iceberg、开放表格式 | 低成本存储、统一批流、开放访问 | 数据工程、算法、分析 | 语义和治理仍分散 |
| 语义分析阶段 | 指标平台、Metric Layer、Semantic Layer | 指标口径、维度、可复用数据集 | 分析师、BI、业务 | 使用门槛仍高 |
| AI 增强阶段 | NL2SQL、ChatBI、Dashboard 解读、报告生成 | 自然语言问数、图表、摘要 | 业务用户 | 容易幻觉、越权、不可追溯 |
| Agentic Data Platform 阶段 | Agent Runtime、Tool Calling、MCP、Run Trace、Eval | 可行动、可审计、可复用的分析能力 | 人 + Agent + 业务系统 | 需要新治理和组织机制 |

这条演进线说明：大数据平台的核心资产从“表和报表”扩展为“数据 + 语义 + 知识 + 技能 + 运行证据”。

## 4. 企业数据智能范式的变化

### 4.1 旧范式：报表交付

```text
业务提需求
-> 数据部门取数 / 建表 / 做报表
-> 业务看报表
-> 再提新需求
```

这种模式的问题是：需求流转慢、分析上下文丢失、分析方法难复用、知识沉淀依赖个人经验。

### 4.2 过渡范式：BI + ChatBI

```text
业务用户选择数据集 / 看板
-> 自然语言问数
-> 返回表格 / 图表 / 简短解释
```

它降低了问数门槛，但如果没有数据集学习、指标语义、权限治理和运行评测，仍然会停留在“能问但不稳定”的阶段。

### 4.3 新范式：Data Agent + 可信数据智能底座

```text
大数据团队沉淀 AI-ready 数据资产和分析能力
-> 业务用户通过 BI、飞书、个人 Agent、多 Agent 工作流自助消费
-> 高频问题、优质答案、失败样例沉淀为模板、Skill、知识库和评测集
```

新范式下，业务用户不是绕开大数据团队，而是通过大数据团队提供的可信底座获得更高自由度。

这个案例非常重要，因为它代表了业务分析师视角的自然诉求：

- 他们希望保留自己的 Obsidian / 飞书 / 文档知识。
- 他们希望可以同时调用企业数据库和个人知识。
- 他们希望多 Agent 帮忙生成分析、报告、洞察和任务。
- 他们希望结果能进入飞书协作和 Quick BI 展示。

但从大数据部门视角看，这张图里真正需要平台提供的不是“再做一个 Agent”，而是让所有 Agent 可安全调用的底座：

| 分析师图中的需求 | 大数据部门应提供的底座能力 |
|---|---|
| 读取企业数据库 | 数据源接入、数据目录、权限账号、查询安全 |
| 结合个人知识 | 知识接入、权限隔离、上下文绑定、引用溯源 |
| 多 Agent 协作 | Agent Runtime、任务编排、工具注册、handoff |
| 输出飞书文档 | 文档导出 Skill、分享鉴权、审计 |
| 输出 Quick BI | BI Adapter、图表资产、数据集发布 |
| 持续反馈迭代 | Run Trace、Eval、反馈回流、模板沉淀 |

所以这张图可以作为企业 Data Agent 建设的用户侧案例，但不能直接作为平台架构终局。平台终局应在它背后补齐数据、语义、治理、安全、评测和资产闭环。

## 6. 大数据团队在 AI 时代的重新定义

### 6.1 从交付报表到交付分析能力

大数据团队过去常被定义为“取数、建模、报表、数仓、治理”团队。AI 时代，这个定义需要升级：

> 大数据团队是企业智能分析能力提供方，负责把企业数据、指标口径、业务知识和分析方法封装成可被人和 Agent 安全消费的服务。

### 6.2 未来要提供的七层资产

| 层级 | 资产 | 说明 |
|---|---|---|
| L0 | 数据连接与原始数据 | 数据源、Lakehouse、API、文件、实时流 |
| L1 | 标准表与领域宽表 | 稳定刷新、标准字段、主题域模型 |
| L2 | 可问数据集 | 字段含义、别名、质量画像、推荐问题、可问范围 |
| L3 | 指标与语义 | 指标口径、维度、业务术语、时间口径、默认过滤 |
| L4 | 分析 Skill | 问数、解读、归因、预测、报告、导出、通知 |
| L5 | 分析产物 | 图表、洞察卡、报告素材、引用关系、历史案例 |
| L6 | 运行反馈 | 准确率、成功率、延迟、成本、用户反馈、评测集 |

### 6.3 团队角色变化

| 传统角色 | AI 时代升级方向 |
|---|---|
| 数据工程师 | 数据产品工程师：负责可问数据集、实时数据、质量、数据服务 |
| 数仓建模师 | 语义建模师：负责指标口径、业务术语、语义视图、维值映射 |
| BI 开发 | 智能分析产品工程师：负责 ChatBI、Dashboard 解读、报告素材 |
| 数据分析师 | Analysis Skill Designer：把分析套路沉淀为可复用 Skill |
| 数据治理 | Agent Governance Owner：治理数据访问、工具调用、Agent 行动和审计 |
| 平台运维 | AgentOps / DataOps：负责 trace、eval、成本、模型路由、回归测试 |

## 7. 推荐建设蓝图

### 7.1 平台公式

```text
Enterprise Data Agent Platform
= Agent Runtime
+ Data Center
+ Dataset Learning Center
+ Semantic & Metric Center
+ Knowledge Center
+ Tool & Execution Hub
+ Skill Hub
+ Asset Hub
+ Governance & Security
+ Run Trace & Eval
+ Integration & Adapter Framework
+ App Matrix
```

### 7.2 应用矩阵

```text
App Matrix
= ChatExcel
+ ChatData / ChatDB
+ ChatBI
+ ChatDashboard
+ ChatReport
+ Data Steward Studio
+ Skill & Asset Hall
+ Domain Analysis Agents
```

### 7.3 第一阶段优先级

第一阶段不要追求完整企业级平台，而要验证“公共能力能被多个应用共同消费”：

1. Data Center 轻量版：统一 Dataset 对象，承接 Excel、DB、BI 数据集。
2. Dataset Learning：字段学习、画像、推荐问题、可问范围。
3. Personal Knowledge：用户上传个人分析知识，用于当前会话或数据集。
4. ChatExcel + ChatBI 双入口：共同消费 Dataset、Knowledge、QueryRun。
5. SQL Guard：只读、限行、超时、敏感字段、审计。
6. Run Trace & Eval：记录问题、SQL、工具调用、结果、反馈。
7. Feishu / BI Adapter：让报告、洞察和图表进入业务协作场景。

### 7.4 中长期路线

| 阶段 | 目标 | 核心能力 |
|---|---|---|
| MVP | 验证双入口和公共数据资产 | ChatExcel、ChatBI Adapter、Dataset Learning、Personal Knowledge、QueryRun |
| P1 | 形成企业分析平台雏形 | 语义中心、Skill Hub、Asset Hub、Data Steward Studio、Dashboard 解读 |
| P2 | 进入经营分析闭环 | ChatReport、洞察归因、报告素材、飞书/BI 协作、评测回归 |
| P3 | 成为企业 Data Agent Core | 多租户、模型网关、AgentOps、MCP 工具治理、行业 Agent |

## 8. 对当前企业的落地建议

### 8.1 不要只做一个聊天框

聊天框只是入口。真正的企业级能力在入口背后：

- 数据集是否可问。
- 指标是否可信。
- 权限是否继承。
- SQL 是否可控。
- 结果是否可追溯。
- 反馈是否能回流。
- 分析过程是否能复用。

### 8.2 不要让 ChatBI 承担所有平台职责

ChatBI 应定位为企业正式数据源上的专业问数引擎，而不是统一资产中心。平台公共资产应放在 Data Agent Platform：

- Dataset。
- DatasetField。
- DatasetProfile。
- QuestionTemplate。
- PersonalKnowledge。
- QueryRun。
- Artifact。
- Skill。
- EvalCase。

### 8.3 不要把个人知识和企业语义混在一起

个人知识适合增强某次分析或某个用户的上下文；企业语义必须经过治理、版本和审批。两者可以相互转化，但不能直接混用。

### 8.4 从“可问数据集 + 运行留痕”开始

如果第一天没有 QueryRun 和 Eval，后续就无法判断准确率；如果没有 Dataset Learning，模型只能猜字段和口径；如果没有 SQL Guard，生产风险不可控。

## 9. 最终判断

Snowflake Summit 26 给出的行业答案是：企业 AI 不会脱离数据平台单独存在，数据平台也不会停留在仓库和 BI。未来的竞争焦点会转向：

```text
谁能把企业数据变成 AI-ready
谁能把业务语义变成 Agent-ready
谁能把工具行动变成 governed action
谁能把分析过程变成可追踪、可评测、可复用资产
```

对大数据团队而言，这意味着角色升级：

> 从“数据和报表交付团队”升级为“企业智能分析能力提供方”。

对业务分析师而言，这意味着工作方式升级：

> 从“提需求、等取数、写 PPT”升级为“带着个人知识和业务判断，调用企业可信数据智能底座，与 Agent 协作完成问数、解读、归因、报告和行动分发”。

这两者不是替代关系，而是新的分工关系：业务侧拥有问题、判断和场景；大数据团队提供可信数据、语义、工具、治理、评测和资产沉淀。
