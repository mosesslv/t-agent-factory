1. Business scenario lens | 用户类型 × 分析生产方式

未来企业数据分析不是单一 BI 入口，而是三类人群共同生产和消费分析资产

战略上应把 Data Agent 放在“数据资产、分析知识和业务入口”之间：让企业公民即时探索，让超级个体生产新资产，让管理层和团队稳定消费高质量结论。

企业公民

一线业务、运营、财务、人事等普通业务用户；重点是低门槛、自助、快速验证。
即时探索用 ChatExcel / 问表能力上传文件或选择业务表，快速问趋势、排行、异常和字段统计。
次抛型数据产品AI Coding 或低代码生成一次性 BI 看板、临时图表和专题分析，用完即沉淀为样例。
进入治理流程高频问题和高价值文件再升级为部门模板、黄金数据集或固定看板。

超级个体

懂业务、懂数据、会用 AI 工具的分析骨干；重点是把个人能力放大为部门级资产。
资产摄取把文件、数仓宽表、黄金数据集、指标、历史报告和分析知识组合进一个任务。
创作新产品生成分析 Agent、报告模板、领域看板、洞察卡和可复用的数据分析流程。
服务他人把自己的分析方法变成团队可调用的能力，而不是停留在个人经验。

消费者

管理层、团队负责人和跨部门协作者；重点是可信、稳定、可解释的经营结论。
固定消费消费已经固化的经营看板、例行报告、订阅摘要和管理层材料。
按场景追问在 T-Claw / PI-Claw、ChatBI、小Q式入口里追问原因、目标差异和行动建议。
形成共识从一次性结论走向统一口径、统一指标和企业级分析知识。
业务抽象从“谁有 BI 权限”转为“谁能生产、复用和消费分析资产”。
建设重点P0 先打通企业公民和超级个体的高频闭环：问数、问表、知识库和报告初版。
承接关系下一页再落到数据对象 × 场景层级，明确哪些资产先进入 Data Agent。
Key message: Data Agent 的战略价值，是把分散的个人探索转化为可复用、可治理、可消费的企业分析资产 1 / 7

4. 能力矩阵 | 产品与资产分工

5. Platform product scope | Data Agent 产品边界

Data Agent 是连接数据资产与业务入口的中间平台，让分析能力可用、可信、可复用

向下接入数仓、指标、知识和 BI 资产；向上服务 T-claw、ChatBI、小Q式入口和管理报告。自身聚焦把问数、解读、洞察、报告做成可运营的企业服务。

向下连接
数据平台 / 数仓物理表、主题宽表、数据质量、血缘、刷新规则。
指标平台指标定义、口径版本、维度层级、计算规则。
知识库 / 模板库业务术语、SOP、诊断框架、历史报告。
BI 资产看板、图表、报表门户、筛选上下文。
Data Agent 提供的六类平台能力
接得上统一接入数据、指标、知识、图表和文件，避免分析入口各自接一套。
问得准把字段别名、指标口径、业务术语和样例问题带入问数过程。
跑得稳组织问数、解读、洞察、报告的多步骤流程，并记录状态。
管得住控制只读、限量、超时、敏感字段、权限和审计，不绕过安全规则。
留得下沉淀结果、图表、洞察、报告、引用关系和人工反馈，形成资产。
评得清持续看准确率、成功率、响应时间、成本和用户反馈，支撑迭代。
Data Agent 不重做数仓、不重做 BI、不替代 T-claw；它负责对外提供多种形态的统一数据分析服务和分析必须的知识体系（指标定义、分析思路、黄金数据集等）。
服务接口
T-Claw / PI-Claw个人级 / 部门级智能入口、身份、对话、任务分发和消息触达。
ChatBI / 小Q式入口自然语言问数、多轮追问、结果解释和交互体验。
FineBI / 看板门户稳定报表、经营看板、图表展示和门户分发。
报告生成接口面向月报、专题分析和管理汇报，提供生成、审批和导出能力。
边界 1：数据和口径仍归源头系统数仓负责数据正确，指标平台负责口径正确，知识库负责业务解释。
边界 2：体验入口可多样化T-claw、ChatBI、小Q和 BI 门户都可以调用同一套 Data Agent 能力。
Architecture principle: Data Agent 做中间能力层，清晰连接上下游系统，而不是重做所有企业数据产品 5 / 7
6. DB-GPT current fit | 能力对标与建设重点

T-Agent 已具备运行时雏形，但缺数据资产层和可信运行层；建设重点应该落在语义完备的黄金数据集体系建设、分析思路知识化、运行过程可审计检测

当前 DB-GPT 对标 QuickBI 小Q整体覆盖约 22%。可复用的是连接、Scene、LLM、SQL 和 Dashboard 图表；差距集中在 Dataset、Metric、Run、Trace、Artifact、Insight 和治理能力。

30%整体覆盖率对标小Q“问数、解读、洞察、报告”的综合能力覆盖。
35%ChatExcel 问数已有上传、DuckDB 和 NL2SQL，缺资产化、质量画像和语义配置。
25%ChatDB / Data已有 schema 问数和 SQL 执行，缺 SQL 安全和数据集层。
20%ChatDashboard / 报告已有图表生成，缺多章节叙事和报告任务化；mvt_demo 是突破口。
可复用数据源连接、Chat Scene、LLM 调用、Excel/CSV 到 DuckDB、SQL 执行、Dashboard 图表数据、mvt_demo 4 章报告链路。
保留
关键缺口DataSource/DataSet/DataField、MetricCatalog、QuestionConfig、Skill Registry、Run/Event/Trace/Artifact。
P0/P1
安全风险ChatWithDbAutoExecute 仍偏裸 SQL 执行，缺 AST 校验、只读限制、限行、超时和审计。
P0
体验差距当前用户流程是“选择库表 → 提问 → SQL → 结果”，缺字段质量、业务口径、反问澄清、结果解读。
P1
差异化场景深度经营报告、私有化部署、模型自由度是 DB-GPT 路线相对 QuickBI 的重点机会。
聚焦
重点建设场景 1：基于黄金数据集的可信问数围绕企业常用问题，先把问数做准、做稳、做可解释。
重点建设场景 2：分析知识驱动的分析报告产出新增分析知识库，让 Agent 在每次生成分析报告时主动提供分析思路，进一步驱动大模型展开数据和经营分析。
重点建设场景 3：运行元数据沉淀与自学习闭环把每次分析运行的元数据信息记录并落库，结合 Eval 体系持续提升 Data Agent 在不同领域的分析能力和知识体系，实现自我提升与自学习。
Strategic answer: DB-GPT 不是小Q替代品；更适合作为开放可控的 Data Agent Runtime 6 / 7
