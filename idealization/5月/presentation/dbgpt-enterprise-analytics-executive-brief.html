2. Capability matrix

面向企业分析需求，大数据部门应提供“数据资产 + 分析 Skills + 平台底座 + BI 工具”的组合能力

ChatBI、小Q、FineBI 不是彼此替代关系；DB-GPT 的机会在于成为开放可控的 Data Agent Runtime，把企业数据资产 Agent 化消费。

数据资产

  • 黄金数据集：领域宽表、字段语义、质量画像
  • 指标体系：指标、维度、口径、时间粒度
  • 业务知识：术语、SOP、诊断框架、样例问题

分析 Skills

  • 问数 Skill：NL2SQL / OLAP / 图表输出
  • 解读 Skill：趋势、异常、解释、引用
  • 报告 Skill：模板、章节、图文、审核

底座平台

  • 连接、模型路由、Skill 编排、运行态
  • SQL 安全、权限、审计、成本控制
  • Run / Event / Trace / Artifact / Feedback

消费工具

  • ChatBI / 小Q：自然语言分析入口
  • FineBI：传统 BI、看板、报表门户
  • DB-GPT：Data Agent runtime 和生态编排
能力域 大数据部门资产 DB-GPT / Data Agent 底座 ChatBI / 小Q类工具 FineBI / 传统 BI 分工逻辑

3. DB-GPT strategic answer

DB-GPT 可作为企业 Data Agent 底座，但距离小Q式企业分析平台仍缺“资产层”和“可信运行层”

当前 DB-GPT 的强项是运行时框架；真正要补的是数据集语义、问数配置、洞察/报告资产和安全治理。

因此建设策略不是“配置一下 DB-GPT”，也不是“重做一个 BI”，而是围绕数据对象和分析任务补齐 Agent 化消费能力。

可复用 数据源连接、Chat Scene、LLM 调用、SQL 执行、Excel 到 DuckDB、Dashboard 图表数据、mvt_demo 报告链路。
缺口 Dataset、Metric、Question Config、SQL Safety、Insight、Report Run、Artifact、Skill Registry。
定位 作为开放可控的 Data Agent Runtime，集成 ChatBI / DataClaw Skills / 小Q式能力和 FineBI 报表资产。
边界 不替代数仓建模，不替代传统 BI 的强可视化编辑，不把所有长链路能力塞进单一 Chat API。
Evidence: pilot/scene, pilot/openapi, pilot/connections, pilot/scene/chat_dashboard, mvt_demo 4 / 6
4. Target architecture

目标架构应从“Scene Router”演进为“数据对象 + 分析 Runtime + Skill 编排”的三层体系

报告生成和洞察归因是多阶段任务，不能只依赖 `/v1/chat/completions`；需要 Run、Event、Trace、Artifact 等运行态对象。

数据对象层

FileDataset：Excel/CSV、schema、profile、版本
PhysicalTable：库表、字段、采样、连接信息
SemanticDataset：JOIN、别名、可问范围
MetricCatalog：指标、维度、口径、时间粒度
Dashboard / Report Asset：看板、章节、版本、审核

分析 Runtime

AskRun：意图、选数、SQL、执行、图表、解释
InsightRun：解读、异常检测、归因、洞察卡
ReportRun：章节编排、模板、素材、产物
SqlSafetyGateway：只读、限行、超时、白名单、审计
ArtifactService：Markdown、JSON、图表、prompt/response trace

Skill 与生态层

Skill Registry：ChatBI、DataClaw、小Q式技能注册
Skill Router：根据意图和数据对象选择能力
Template Manager：问数、解读、报告、归因模板
Evaluation：标准问题、成功率、成本、延迟
Integration Layer：FineBI、指标平台、消息/任务系统
Architecture principle: 先任务服务化和安全边界，再逐步引入 AWEL/多 Agent 编排 5 / 6
5. Roadmap and first moves

建设路径：4-6 周形成可演示闭环，3-4 个月形成平台雏形，再进入企业级核心增强

Phase 0

3-5 人天 | 方案冻结
  • 明确目标演示数据和高频问题
  • 锁定依赖和新服务隔离策略
  • 定义 Dataset / AskRun / ReportRun 对象草案

Phase 1

4-6 周 | MVP 闭环
  • Chat Excel 上传问数
  • Chat Data/DB 生成 SQL、预览/编辑
  • fin-report JSON 到 4 章报告服务化
  • 最小 SQL 安全网关

Phase 2

6-10 周 | 平台雏形
  • SemanticDataset 和问数配置
  • 业务术语、样例问题、few-shot
  • Dashboard 解读和报告编辑
  • Skill 路由和素材联动

Phase 3

8-12 周 | 企业增强
  • MetricCatalog 和指标树归因
  • 异常检测、贡献度拆解、洞察卡
  • 真实业务模型计算和报告模板库
  • 审计、权限、导出、订阅
64-95 人天MVP：3-5 人小队，形成可演示主线
182-283 人天P1：5-7 人小队，形成平台雏形
365-585 人天P2：企业级核心增强，进入生产化
First-wave modules: DatasetService, SqlSafetyGateway, AskRunService, ReportRunService, ArtifactService, Skill Registry 6 / 6
