# v2.0 需求收敛与版本边界

## 1. v2.0 定位

v2.0 定位为 **平台化试点版本**，周期按 2026 年 7-9 月、约 8 周规划。

v2.0 不定义为完整企业级平台，也不只是 v1.0 的单点功能增强。它的核心目标是：

> 在 v1.0 双入口能力基础上，把 ChatExcel、ChatBI、Chat 报告和可对外提供的分析 Skills 串成一条可试点的企业数据分析平台主链路。

## 2. v1.0 基础

v1.0 已经形成的基础能力包括：

- ChatExcel 和 DB-GPT 页面内 ChatBI 双入口。
- 数据集学习。
- 个人分析知识上传。
- StarRocks 选库选表和 PSI 周/月宽表作为 MVP 数据集。
- ChatBI 与 ChatExcel 基于同一批数据集资产进行问数/问表。
- 运行留痕和基础反馈。

v2.0 的重点不是推翻 v1.0，而是把这些能力平台化：

- 数据源和数据集从“能选表”升级到“可管理、可展示、可复用”。
- 数据集学习从“应用内辅助”升级到“黄金数据集语义资产”。
- 个人知识从“当前会话增强”升级到“个人/企业知识隔离与可控使用”。
- 问数/问表从“双入口体验”升级到“ChatExcel、ChatBI、Chat 报告、Skills 共同消费的平台能力”。
- 报告从“生成初版”升级到“深度经营分析报告主链路”。

## 3. 试点资源边界

当前确认资源：

| 角色 | 投入 |
|---|---:|
| 后端研发 A | 80% |
| 后端研发 B | 50% |
| Agent 研发高级 | 1 人 |
| Agent 研发初级 | 2 人 |
| AI 产品 | 1.5 人 |
| AI 产品负责人 | 1 人 |
| 全栈研发 | 可额外申请，但默认不具备算法能力 |

资源判断：

- 8 周内可以交付一条平台化试点主链路。
- 不适合同时深做完整数据源管理、完整知识平台、完整权限平台、完整模型治理、完整飞书协同。
- Agent 研发资源相对充足，应把优势放在深度经营分析报告链路、分析计划、工具编排、过程可视和 Skills 化。
- 后端资源相对紧，需要控制数据源、权限、分享和管理后台的深度。

## 4. v2.0 主线

v2.0 主线建议定义为：

> **基于黄金数据集、知识和权限的销售经营分析平台化试点。**

主链路：

```text
进入 DB-GPT / T-Agent 平台
  -> 选择销售经营黄金数据集
  -> 选择 ChatExcel / ChatBI / Chat 报告 / Skill 入口
  -> 系统加载数据集语义、权限、知识上下文和模型配置
  -> 用户发起问数、问表或报告议题
  -> Agent 进行议题理解、分析计划、工具编排和过程展示
  -> 用户查看中间分析、必要时调整分析步骤或代码
  -> 生成销售经营分析报告
  -> 输出为飞书文档或分享到飞书群
  -> 平台记录运行过程、权限命中、数据引用、知识引用和分享记录
```

## 5. 首批业务场景

首批业务场景选择：

> **销售经营分析报告。**

建议进一步收敛为 1-2 个可验收样例：

1. 销售经营周报。
2. 销售经营月报或销售异常归因专题。

原因：

- 销售经营场景高频、管理层容易理解。
- 可以复用 PSI 周/月粒度宽表或销售经营宽表。
- 适合验证 ChatBI 问指标、ChatExcel 问表、Chat 报告生成和 Skills 编排。
- 适合飞书输出和群分享。

## 6. 黄金数据集定义

v2.0 中“黄金数据集”不只是认证宽表，也不只是 BI 语义数据集。

建议定义为三层：

| 层级 | 定义 | v2.0 是否纳入 |
|---|---|---|
| L1 认证宽表 | 已确认来源、负责人和权限边界的 StarRocks 宽表 | 纳入 |
| L2 宽表 + 语义口径 | 在认证宽表基础上补充字段解释、指标口径、同义词、样例问题、可问范围 | 纳入 |
| L3 可支撑报告的数据产品 | 能直接支撑固定经营报告模板，包含业务说明、报告适用范围、默认分析维度和质量说明 | 试点纳入 |

因此 v2.0 黄金数据集建议定义为：

> 经平台认证、带语义口径、可被 ChatExcel、ChatBI、Chat 报告和 Skills 共同消费，并能支撑销售经营分析报告的数据产品。

## 7. 需求域收敛

### 7.1 数据源连接与数据集管理

v2.0 做：

- StarRocks 数据源连接管理。
- 销售经营相关宽表注册。
- 黄金数据集列表展示。
- 数据集详情展示：字段、字段解释、指标口径、样例问题、负责人、更新时间。
- 数据集学习后台管理：学习状态、学习结果、人工修正入口。

v2.0 不做或弱做：

- 多数据库连接器泛化。
- 复杂连接凭证生命周期。
- 数据血缘全链路。
- 完整数据质量平台。

### 7.2 大模型选择

v2.0 做：

- 后台模型配置。
- 按任务类型配置默认模型。
- 支持至少 2-3 个模型配置。
- 记录模型调用任务、耗时、失败原因。

建议任务类型：

| 任务 | 模型策略 |
|---|---|
| 问数 / SQL 生成 | 稳定、低延迟模型 |
| 字段学习 / 语义补全 | 成本可控模型 |
| 深度分析计划 | 强推理模型 |
| 报告撰写润色 | 长文本质量较好的模型 |
| 知识检索摘要 | 低成本摘要模型 |

v2.0 不做或弱做：

- 普通用户自由切换模型。
- 自动模型路由优化。
- 多模型 A/B 评测平台。
- 完整成本治理。

### 7.3 深度经营分析报告

v2.0 做：

- 议题理解。
- 分析计划制定。
- 分析引擎编排。
- 数据分析过程可视。
- 展示 SQL、代码、分析片段、意图和中间结论。
- 支持常规模式和 Deep Research 深度分析模式。
- 支持联网开关。
- 支持更深度知识库检索。
- 支持报告撰写和润色。
- 支持报告输出到飞书文档。

建议把报告链路拆成：

```text
议题输入
  -> 议题理解
  -> 确认分析目标和数据范围
  -> 生成分析计划
  -> 执行问数/问表/检索/联网/计算工具
  -> 展示过程和中间结论
  -> 用户调整分析步骤或代码
  -> 二次执行
  -> 生成报告大纲
  -> 撰写报告
  -> 润色与格式化
  -> 输出飞书
```

v2.0 需要谨慎控制：

- 代码可编辑建议先面向分析师/高级用户。
- 普通用户可先看过程、改分析目标、改维度、改时间范围。
- 不建议第一期做完整 Notebook。

### 7.4 个人/企业知识库隔离

v2.0 做：

- 个人知识库。
- 企业知识库。
- 个人知识默认仅本人可见。
- 企业知识作为平台级只读或受控知识源。
- 报告生成时明确标记使用了哪些知识来源。
- 检索时按用户身份和权限过滤。

v2.0 暂不完整做：

- 领域级知识审批流。
- 企业知识版本治理。
- 知识市场。
- 复杂知识空间。

建议隔离规则：

| 知识类型 | 可见范围 | 典型内容 | v2.0 用法 |
|---|---|---|---|
| 个人知识 | 本人，当前会话可共享 | 用户上传 Word、飞书文档、个人分析说明 | 辅助个人问数和报告生成 |
| 企业知识 | 平台配置，可按权限读取 | 指标口径、业务术语、销售分析框架、历史标准报告 | 支撑销售经营报告和统一口径 |

### 7.5 权限控制

v2.0 权限源：

> 依据 ChatBI 的权限表作为基础，再在 T-Agent / DB-GPT 平台上集成。

建议理解为：

- ChatBI 权限表是数据权限和业务数据访问的基础源。
- T-Agent / DB-GPT 负责把 ChatBI 权限映射到平台对象：数据源、数据集、报告、知识、分享。
- 后续可抽象为统一鉴权适配层。

v2.0 做：

- 用户身份映射。
- 数据集可见性鉴权。
- 查询执行前鉴权。
- 报告生成前鉴权。
- 报告查看和分享时鉴权。
- 权限命中记录。

待审计：

- ChatBI 权限表覆盖粒度：用户、角色、组织、数据集、库表、字段、行权限。
- ChatBI 用户与 DB-GPT 用户是否同源。
- ChatBI 权限是否可被平台实时查询。

### 7.6 分享、转发和飞书集成

v2.0 飞书目标：

1. 读取飞书文档作为知识。
2. 输出报告为飞书文档。
3. 分享到飞书群。

v2.0 做：

- 飞书文档正文读取。
- 报告生成后创建飞书文档。
- 报告链接或摘要分享到飞书群。
- 分享记录留痕。

v2.0 需要明确安全策略：

- 分享前是否重新鉴权。
- 飞书群成员是否需要逐个鉴权。
- 报告是动态链接还是静态快照。

建议 MVP 先做：

> 受控静态报告输出 + 分享动作留痕。

不建议第一期做：

- 飞书审批流。
- 飞书评论协作闭环。
- 飞书组织权限自动同步。
- 报告动态权限继承的复杂模型。

## 8. Skills 化方向

v2.0 需要让 ChatExcel、ChatBI、Chat 报告等能力可以对外提供 Skills。

建议第一批 Skills：

| Skill | 能力说明 | 消费场景 |
|---|---|---|
| DatasetSelectSkill | 选择和读取黄金数据集 | ChatExcel、ChatBI、报告 |
| MetricQuerySkill | 基于 ChatBI 权限和语义问指标 | ChatBI、报告 |
| TableAnalysisSkill | 基于表数据做明细分析和总结 | ChatExcel、报告 |
| KnowledgeRetrievalSkill | 检索个人/企业知识 | ChatExcel、ChatBI、报告 |
| SalesReportSkill | 生成销售经营分析报告 | Chat 报告 |
| FeishuExportSkill | 输出报告到飞书文档或群 | 报告、分享 |

Skills 化的意义：

- 让 ChatExcel、ChatBI 和 Chat 报告不再各自实现能力。
- 让深度报告可以编排已有能力。
- 为后续 Data Agent 市场或 Skill Hub 做准备。

## 9. v2.0 建议版本边界

### v2.0 必须做

- 销售经营黄金数据集展示与选择。
- StarRocks 销售经营宽表接入。
- 数据集学习后台管理。
- ChatBI 权限表集成的最小鉴权。
- 后台模型配置。
- ChatExcel / ChatBI / Chat 报告共用数据集和知识上下文。
- 深度经营分析报告主链路。
- 常规模式和 Deep Research 模式。
- 飞书文档读取。
- 报告输出为飞书文档。
- 分享到飞书群。
- 运行过程、知识引用、数据引用、权限命中、分享记录留痕。
- 第一批对外可调用 Skills。

### v2.0 可选做

- 代码编辑后重跑。
- 报告模板管理。
- 企业知识维护后台。
- 多模型调用成本统计。
- 报告版本记录。
- 分享链接权限控制。

### v2.0 不建议做

- 完整多租户权限体系。
- 完整企业知识库审批发布。
- 多数据源连接器平台。
- 完整 BI 语义层重建。
- 完整 Notebook 式代码编辑器。
- 完整模型治理、自动路由和 A/B 测试。
- 飞书审批和评论协同闭环。

## 10. v2.0 关键依赖

| 依赖 | 影响 |
|---|---|
| ChatBI 权限表结构 | 决定数据集、查询、报告分享如何鉴权 |
| 销售经营宽表字段和口径 | 决定黄金数据集和报告质量 |
| 飞书开放平台权限 | 决定能否读取文档、创建文档、群分享 |

| 销售经营报告样例 | 决定报告模板、验收标准和 Agent 分析计划 |
| Skills 调用协议 | 决定能力是否能对外复用 |

## 11. 下一轮必须确认的问题

1. 销售经营分析报告的第一版样例是周报、月报，还是异常归因专题？
2. 销售经营黄金数据集具体是哪几张 StarRocks 宽表？
3. 黄金数据集是否需要管理员维护页面，还是先从配置文件/后台初始化？
4. ChatBI 权限表覆盖哪些权限粒度？
5. DB-GPT / T-Agent 用户与 ChatBI 用户如何映射？
6. Deep Research 模式允许联网访问哪些来源？
7. 代码可编辑本期是必做、可选，还是只做过程可视？
8. 飞书群分享是否需要校验群成员权限？
9. 报告输出飞书文档后，是静态快照还是可回到平台重新分析的链接？
10. Skills 对外提供给谁调用：平台内部 Agent、其他业务系统，还是用户自定义 Agent？

## 12. 当前建议

v2.0 应按 8 周试点版本规划，主线聚焦：

> **销售经营黄金数据集 + 深度经营分析报告 + 权限集成 + 飞书交付 + Skills 化。**

这条主线能把当前设想中的 6 个方向串起来：

- 数据源连接和数据集管理服务于黄金数据集。
- 模型选择服务于不同分析任务。
- 深度经营分析报告是前台核心体验。
- 企业/个人知识隔离保障口径可信。
- ChatBI 权限表集成保障数据安全。
- 飞书读写和群分享完成协作交付闭环。
