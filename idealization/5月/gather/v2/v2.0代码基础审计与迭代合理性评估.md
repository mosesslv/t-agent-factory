# v2.0 代码基础审计与迭代合理性评估

版本：v0.1  
日期：2026-05-28  
状态：基于当前 DB-GPT 代码扫描的规划校准稿  
代码范围：`pilot/`、`mvt_demo/`、`docs/dev/v1.0/solution/`

## 1. 结论

v2.0 当前方向总体合理，但需要调整表达和建设优先级。

当前 DB-GPT 代码已经具备：

- ChatExcel 文件分析和字段学习雏形。
- ChatDB / ChatDashboard 的数据库问数和图表报告雏形。
- 数据源连接管理。
- 知识库/RAG 空间、文档、切片、向量检索雏形。
- LLM worker 列表、启动、停止等基础管理能力。
- AWEL 工作流底层框架。
- Agent Command / Plugin 形式的工具注册雏形。
- mvt_demo 中的经营报告章节编排 MVT。

但当前代码缺少 v2.0 要成为平台级试点所必需的中间层：

- 没有平台级 Dataset / GoldenDataset 资产模型。
- 没有 ReportArtifact / AnalysisRun / StepRun / Evidence 资产模型。
- 没有权限适配层，现有 `user_name`、`sys_code` 只能做轻量隔离，不能承担企业鉴权。
- 没有 StarRocks 原生连接器，若 StarRocks 走 MySQL 协议可短期复用 MySQL；否则需要新增连接器。
- 没有 ChatBI Adapter 和 ChatBI 权限表适配。
- 没有飞书读写集成。
- 没有可对外稳定调用的 Skill Contract。
- 深度报告链路还停留在 `mvt_demo` 离线章节编排和 `ChatDashboard` 图表组合，尚未平台化。

因此 v2.0 最合理的定位不是“完整平台”，而是：

> **平台化试点版本：补齐数据集资产、运行资产、权限适配、报告编排和 Skills Contract，让 ChatExcel、ChatBI、Chat 报告可以共用平台能力。**

## 2. 代码现状扫描

### 2.1 ChatExcel

相关代码：

- `pilot/scene/chat_data/chat_excel/excel_learning/chat.py`
- `pilot/scene/chat_data/chat_excel/excel_analyze/chat.py`

现状：

- `ExcelLearning` 会读取样例数据，交给 LLM 生成结构理解结果。
- `ChatExcel.prepare()` 会在首次会话中触发 `ExcelLearning`。
- `ChatExcel` 使用 DuckDB SQL 做表格分析，并通过 `ApiCall.display_sql_llmvis` 展示结果。

判断：

- 适合继续作为“问表、明细分析、分析总结”的入口。
- 但现有学习结果主要存在会话消息里，不是可复用资产。
- v2.0 若要支撑黄金数据集，不能继续只依赖 ChatExcel 会话内学习，需要把字段学习结果落到 Dataset / DatasetField / DatasetProfile。

### 2.2 ChatDB / ChatDashboard

相关代码：

- `pilot/scene/chat_db/auto_execute/chat.py`
- `pilot/scene/chat_dashboard/chat.py`
- `pilot/scene/chat_dashboard/data_preparation/report_schma.py`

现状：

- ChatDB 可基于数据库 schema 生成 SQL 并执行。
- ChatDashboard 可基于数据库和 dashboard 模板生成多个图表项。
- `ReportData` 当前主要是图表集合：`conv_uid`、`template_name`、`charts`。

判断：

- ChatDashboard 是报告方向的可复用起点，但它更像“图表生成器”，不是经营分析报告引擎。
- v2.0 不能只扩展 `ChatDashboard` Prompt，否则会缺少议题理解、计划、证据、过程、报告版本、飞书输出。
- 应新增独立的 `analysis_report` 或 `report_agent` 平台模块，再复用 ChatDashboard 的图表能力。

### 2.3 mvt_demo 报告编排

相关代码：

- `mvt_demo/engine/chapter_writer.py`
- `mvt_demo/prompts/fin_report.py`

现状：

- `ChapterWriter` 已复现 fin-report 的四章节报告生成思路。
- 经营总论包含多次 LLM 调用：抽取、综合、润色、整合。
- 当前输入是离线 JSON，不接 DB-GPT 数据源、权限、Dataset、Artifact。

判断：

- mvt_demo 是 v2.0 深度经营分析报告的关键参考。
- 但它不能直接成为平台服务，需要抽象为 ReportPlan、ReportStep、ReportArtifact。
- v2.0 建议复用其章节编排思想，而不是继续在 `mvt_demo` 内堆功能。

### 2.4 数据源连接

相关代码：

- `pilot/connections/manages/connection_manager.py`
- `pilot/connections/manages/connect_config_db.py`
- `pilot/common/schema.py`
- `pilot/openapi/api_v1/api_v1.py`

现状：

- 已有连接配置表 `connect_config`。
- 已有数据库连接增删改查接口。
- 已支持 MySQL、DuckDB、SQLite、MSSQL、PostgreSQL、ClickHouse、Spark 等类型。
- 没有发现 StarRocks 类型。

判断：

- v2.0 做数据源连接和数据集管理是合理的，因为已有连接管理底座。
- 但如果 StarRocks 通过 MySQL 协议访问，短期可配置为 MySQL；如果需要 StarRocks 方言、元数据、权限和 SQL 特性，应新增 `StarRocksConnect` 和 `DBType.StarRocks`。
- 现有连接是“数据库连接”，不是“数据集资产”。黄金数据集不能直接等同于 `connect_config`。

### 2.5 知识库

相关代码：

- `pilot/server/knowledge/space_db.py`
- `pilot/server/knowledge/document_db.py`
- `pilot/server/knowledge/api.py`
- `pilot/server/knowledge/service.py`

现状：

- 有 KnowledgeSpace、KnowledgeDocument、DocumentChunk、向量同步、文档上传、检索等能力。
- `KnowledgeSpaceEntity` 有 `owner` 字段。
- `KnowledgeDocumentEntity` 通过 `space` 归属知识空间。
- 当前权限主要靠查询条件和空间名约定，没有企业级隔离模型。

判断：

- v2.0 的个人/企业知识隔离方向合理，且可复用当前知识库框架。
- 但需要新增 `knowledge_scope`、`visibility`、`source_type`、`owner_user`、`org_code`、`dataset_uid` 等元数据。
- 飞书文档读取不应混在普通上传接口里，应新增 Feishu document connector。

### 2.6 模型选择

相关代码：

- `pilot/server/llm_manage/api.py`
- `pilot/openapi/api_v1/api_v1.py`
- `pilot/scene/base_chat.py`

现状：

- 已有 worker model list/start/stop。
- Chat 请求中可以传 `model_name`。
- `BaseChat` 若未传模型则使用 `CFG.LLM_MODEL`。

判断：

- v2.0 做“大模型后台切换”合理，但不应做成用户前台下拉。
- 代码更适合做任务级模型配置：SQL 生成、字段学习、报告规划、报告撰写、知识摘要分别配置默认模型。
- 需要新增 `model_task_profile` 或类似配置表，而不是只复用 worker 管理 API。

### 2.7 AWEL / Agent / Skill

相关代码：

- `pilot/awel/`
- `pilot/base_modules/agent/commands/command_mange.py`
- `pilot/scene/chat_factory.py`

现状：

- AWEL 有 DAG、Operator、Runner 等底层机制。
- CommandRegistry / ApiCall 可作为工具调用雏形。
- ChatFactory 仍是以 ChatScene 为中心的场景工厂。

判断：

- v2.0 提“对外提供 Skills”方向是合理的，但当前代码没有稳定 Skill Contract。
- 不建议直接把 Skills 做成散落的 Prompt 或 Scene。
- 应定义平台级 Skill 接口：输入 Schema、输出 Schema、权限声明、依赖数据集、运行记录、错误结构。

### 2.8 权限与身份

相关代码：

- `pilot/openapi/api_v1/api_v1.py`
- `pilot/memory/chat_history/*`
- `pilot/server/knowledge/space_db.py`
- `pilot/connections/manages/connect_config_db.py`

现状：

- API 中有 `user_name`、`user_id`、`sys_code`。
- chat history、prompt、plugin、connect config 有部分 user/sys_code 字段。
- 未发现完整 RBAC、组织、角色、数据权限、行列权限实现。

判断：

- v2.0 以 ChatBI 权限表为基础是必要且合理的。
- 但必须新增 `auth_adapter`，把 ChatBI 权限映射到平台对象。
- 权限不应后置，否则数据集展示、知识检索、报告分享、飞书外发都会出现绕权风险。

## 3. 对 v2.0 当前规划的合理性判断

### 合理的部分

1. **以销售经营分析报告作为主线合理。**  
   当前有 ChatDashboard 图表生成和 mvt_demo 报告章节编排，可以组合成试点主线。

2. **数据源连接和黄金数据集管理合理。**  
   现有连接管理只能管 DB 连接，缺 Dataset 层。v2.0 补 Dataset/GoldenDataset 是必须的。

3. **模型后台切换合理。**  
   当前模型支持是 worker 级和请求级，缺任务级模型配置。v2.0 做后台任务模型路由是合理的。

4. **个人/企业知识隔离合理。**  
   当前 KnowledgeSpace 有 owner，但不够。v2.0 补 scope 和权限过滤正好贴合代码现状。

5. **ChatBI 权限表集成合理。**  
   当前 DB-GPT 没有企业级权限，不能自建一套复杂权限后再对接 ChatBI。以 ChatBI 权限为基础更现实。

6. **飞书读写与群分享合理。**  
   v1.0 已明确飞书文档作为知识输入，v2.0 扩展到报告输出和群分享是自然演进。

### 需要修正的部分

1. **“ChatExcel、ChatBI、Chat 报告都跑通”要有公共运行底座。**  
   如果还是按各自 Scene 开发，会重复造上下文、权限、知识检索和运行记录。

2. **“深度经营分析报告”不能直接等于 ChatDashboard。**  
   ChatDashboard 只能生成图表数据；深度报告需要计划、步骤、证据、章节、Artifact、版本。

3. **“Skills 化”不能只靠现有 Agent Command。**  
   CommandRegistry 适合工具调用，不足以承担可对外复用的企业 Skill。

4. **“代码可编辑”不应作为 v2.0 必做。**  
   当前没有 Notebook、沙箱、权限、依赖管理和结果版本。v2.0 可以先做“SQL/代码可查看、可复制、可重跑部分步骤”，代码编辑放可选。

5. **“数据源连接”不应扩大到多源平台。**  
   8 周试点应聚焦 StarRocks/MySQL 协议宽表 + Excel/CSV。多连接器平台后置。

## 4. 建议补充的 v2.0 关键迭代

### 4.1 平台资产模型

必须新增：

- DataSource
- Dataset
- DatasetField
- GoldenDataset
- MetricDefinition
- DatasetQuestion
- DatasetLearningJob

原因：

- 当前只有 DB 连接和 ChatExcel 会话学习，没有资产层。
- 黄金数据集、报告、权限和 Skills 都依赖稳定 Dataset UID。

### 4.2 分析运行模型

必须新增：

- AnalysisRun
- AnalysisStep
- ToolCall / SkillCall
- Evidence
- AnalysisArtifact
- ReportArtifact
- ShareRecord

原因：

- 当前 trace 只能记录 span，不是面向用户和业务验收的分析过程资产。
- 深度报告必须能回看：用了什么数据、什么 SQL、什么知识、什么模型、什么结论。

### 4.3 权限适配层

必须新增：

- ChatBIAuthAdapter
- PermissionSubject
- PermissionObject
- PermissionDecision
- PermissionAuditLog

原因：

- 数据集展示、SQL 执行、知识检索、报告查看、飞书分享都要鉴权。
- 不能把权限判断散落在 ChatExcel、ChatBI Adapter、Report Agent 中。

### 4.4 Skill Contract

必须新增：

- Skill 定义注册表。
- 输入输出 Schema。
- 权限声明。
- 依赖资源声明。
- 运行记录。
- 错误码。

首批 Skills 建议：

- DatasetSelectSkill
- MetricQuerySkill
- TableAnalysisSkill
- KnowledgeRetrievalSkill
- SalesReportSkill
- FeishuExportSkill

### 4.5 报告编排服务

必须新增：

- ReportPlanService
- ReportExecutionService
- ReportWriterService
- ReportArtifactService

原因：

- mvt_demo 证明报告章节编排可行，但不是平台服务。
- ChatDashboard 证明图表可生成，但不是深度报告。
- v2.0 应把二者合并为可追踪的报告 Agent。

### 4.6 Feishu Connector

必须新增：

- FeishuDocReader
- FeishuDocWriter
- FeishuGroupMessenger
- FeishuShareAudit

原因：

- 飞书既是知识来源，也是报告交付渠道。
- 需要独立 connector，不应混入 KnowledgeService 或 ReportService。

## 5. 建议调整后的 v2.0 主线

原主线：

> 销售经营黄金数据集 + 深度经营分析报告 + 权限集成 + 飞书交付 + Skills 化。

建议修正为：

> **销售经营黄金数据集驱动的深度分析报告试点：以 Dataset/AnalysisRun/Skill/Permission 四个公共底座串联 ChatExcel、ChatBI、Chat 报告和飞书交付。**

这样更贴合代码现实：先补平台对象和运行底座，再把现有 Scene 作为消费者接入。
