1. Product planning recap | 平台产品定位

Data Agent 的平台定位，是把数据资产、分析能力和业务入口聚合为可运营的企业智能分析服务

平台不重做数仓、不替代 BI，也不绑定单一入口；它在中间层统一可问数据集、业务口径、分析知识、问数过程、结果产物和运行反馈。

业务入口层用户在哪里使用
ChatBI / 小Q自然语言问数、多轮追问、结果解释。
FineBI / 看板稳定报表、图表门户、经营监控。
T-Claw / PI-Claw个人级或部门级智能入口。
Data Agent 市场应用分发、模板和技能市场。
价值业务用户可以从多入口使用同一套可信分析能力。

2. DB-GPT current fit | 能力差距定位

DB-GPT 已具备运行时雏形，但要对标小Q式问数，必须补齐数据集语义资产和可信运行机制

当前可复用的是连接、Scene、LLM、Excel/CSV 读取、DuckDB 查询和图表展示；核心缺口在 Dataset、Field、Question、Run、Trace、Permission 和 Eval。

4. Zoom-in | 为什么先聚焦 ChatExcel 数据学习

ChatExcel 是最适合做第一阶段样板的模块，因为它既有可复用运行链路，也暴露了平台化数据学习的关键缺口

现有 ChatExcel 可以“跑起来”，但还不能让数据集被持续学习、校准和复用。新增数据学习层后，它可以成为后续 ChatDB、报告和洞察能力的共同底座。

35%当前覆盖上传、抽样、DuckDB、NL2SQL 和图表已有基础。
P0核心缺口没有可管理的数据集语义资产，问数依赖历史消息。
低工程切入风险文件场景优先，暂不触碰复杂数仓权限和跨库查询。
高业务可见度用户可以直接看到字段学习、快捷问题和问数效果提升。
数据集选择从“上传后直接问”升级为“选择已学习数据集或上传生成临时数据集”。
P0
数据集学习识别字段语义、维度度量、日期字段、质量画像和样例问题。
P0
人工校准用户可修改字段注释、同义词、默认聚合、是否可问。
P0
快捷提问初始化用户可以问的问题，并支持编辑、排序、启停。
P0
安全问数学习资产进入 Prompt，SQL 通过只读、字段白名单和限行校验。
P1
运行反馈记录问题、SQL、状态、耗时、错误和用户反馈，支撑后续评估。
P1

5. Functional solution | 功能级解决方案

新增 Dataset Learning 层，把一次性 ExcelLearning 升级为可保存、可编辑、可复用的数据集学习资产

改造原则是不重写 ChatExcel 运行链路，而是在前置学习阶段沉淀数据集语义；ChatExcel 运行时只消费学习产物，并通过 SQL Guard 保证可控执行。

01选择或创建数据集支持 Excel/CSV 上传，生成 `dataset_uid`，记录来源、行列数、状态和采样策略。
02自动学习Profiler 计算类型、分布、空值率、TopN；LLM 生成字段解释和问题模板。
03人工校准业务用户调整字段注释、语义角色、同义词、默认聚合和可问状态。
04引导问数在 ChatExcel 中展示快捷提问，注入字段目录和学习摘要生成 SQL。
05留痕评估保存问题、SQL、命中字段、执行状态、错误和用户反馈，进入迭代闭环。
Dataset数据集名称、来源、类型、状态、所有者。
DatasetField字段类型、注释、维度/度量、同义词、是否可问。
DatasetProfile空值率、唯一值、TopN、样例值、时间范围。
QuestionTemplate快捷问题、意图、字段引用、排序和启停。
LearningJob学习状态、进度、建议、失败原因和重试。
QueryRun问数 SQL、结果状态、耗时、反馈和审计。
保留ExcelReader、DuckDB、LLM 问数和图表渲染链路。
新增数据集资产表、学习任务、字段编辑和问题模板。
改造ChatExcel Prompt 从“读历史”变为“读数据集学习产物”。
Design principle: 学习产物显式化，问数运行受约束，反馈数据可沉淀 6 / 8
6. Architecture zoom-in | 模块架构

技术实现上新增一个隔离的 Dataset Learning 服务层，避免大改 DB-GPT 核心框架

后端新增数据集学习 API、Service、DAO、Profiler 和 SQL Guard；ChatExcel 只在运行时加载 dataset context，前端增加数据集选择、字段编辑和快捷提问配置。

输入资产
Excel / CSV沿用当前 ChatExcel 文件读取和 DuckDB 注册能力。
DB 表 / 视图P1 扩展，只读连接、采样 SQL、超时约束。
业务知识字段口径、术语、同义词、常见问题。
用户校准人工注释、字段角色、启停配置。
Dataset Learning 服务层
Dataset Registry统一管理数据集目录、状态、来源和版本。
Profiler计算字段类型、分布、空值率、样例值和质量提示。
LLM Learner生成字段语义、维度度量识别和快捷问题。
Manual Curation保存用户修订注释、同义词、聚合方式和可问字段。
Prompt Builder把学习产物压缩成 ChatExcel 可用上下文。
SQL Guard执行只读、白名单、限行、超时和审计约束。
消费入口
ChatExcel优先消费学习后的字段目录和快捷问题。
ChatDBP1 复用数据集资产层，连接真实库表。
报告服务引用问数结果、字段口径和分析上下文。
运营评估统计成功率、准确率、成本、延迟和反馈。
工程边界新增 `pilot/server/dataset_learning` 和 `chat_excel/dataset_learning`，少改核心 Scene。
MVP 数据源先 Excel/CSV，降低权限、连接和大数据查询风险。
前端策略优先找回源码；若不可得，先做轻量独立管理页验证闭环。
Implementation posture: 以隔离模块方式增强现有 DB-GPT，而不是重构整个框架 7 / 8
7. Delivery plan | 排期、人力与决策

ChatExcel 4-6 周迭代包建议以 55-80 人天完成，并作为报告 MVP 的前置可信问数能力

本页不是重写平台整体排期，而是展开附件模块表中 ChatExcel 的 4-6 周工作包：第一阶段只承诺单数据集、Excel/CSV 优先、可学习、可编辑、可问数；DB 表、复杂权限、多数据集和指标平台放到后续阶段。

阶段 周期 核心交付 关键角色 用户可见结果
方案和模型 0.5 周 数据模型、API、Prompt 输出格式、验收数据集。 产品、后端、算法 范围冻结，避免 MVP 扩散。
元数据后端 1 周 Dataset / Field / Question / LearningJob / QueryRun。 后端 可创建、查看、预览数据集。
学习任务 1 周 Profiler、LLM Learner、结构化解析、学习状态。 算法、后端 生成字段注释、画像和快捷问题。
校准和问数 1.5-2 周 字段编辑、问题模板、Prompt Builder、SQL Guard。 前端、后端、算法 人工修订后，ChatExcel 使用新语义问数。
测试和演示 0.5-1 周 端到端测试、演示脚本、失败兜底、验收指标。 QA、产品、研发 向报告 MVP 输出可信分析结果。
ChatExcel 4-6 周包：55-80 人天后端 1、前端 1、算法/Prompt 1、产品 0.5、QA 0.5，4-6 周交付。
平台雏形：140-220 人天扩展 DB 表、权限、评估、反馈、前端体验，10-12 周交付。
企业增强：300-500+ 人天多数据集、指标平台、主题域治理、学习加速、报告和洞察联动。
决策事项 推荐选择 原因
MVP 数据源 Excel/CSV 优先 复用现有 ChatExcel，最快形成演示闭环，避开真实数仓权限和性能风险。
多数据集 MVP 不做 先证明单数据集语义学习和问数准确率，多表关联放入 P2。
前端路线 优先找回源码；否则轻量管理页 当前仓库主要是构建产物，直接改 `_next` 不适合作为长期方案。
MVP 验收口径
  • 字段识别覆盖率不低于 90%。
  • 快捷问题可执行率不低于 80%。
  • 常见问数 SQL 成功率不低于 80%。
本轮立项请求

  • 确认第一批演示数据集和业务验收人。
  • 确认是否投入前端源码恢复或轻量管理页。
