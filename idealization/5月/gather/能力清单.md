# 4. L0-A 详细 L1-L2 能力定义

## A1. Portal & Workspace：统一入口与工作区

```text
Portal & Workspace
= 用户入口 + 应用导航 + 项目空间 + 会话上下文 + 分析任务上下文
```

| L2 能力                         | 定义                                          | 说明                                                   |
| ------------------------------- | --------------------------------------------- | ------------------------------------------------------ |
| A1.1 应用门户                   | 提供 ChatExcel、ChatBI、ChatReport 等统一入口 | 避免每个应用独立入口、独立账号、独立上下文             |
| A1.2 Workspace / Project        | 按项目组织数据、知识、会话、报告素材          | 例如“Q2 经营分析项目”“客户流失分析项目”            |
| A1.3 Conversation Session       | 管理用户对话、上下文、历史提问                | 支持多轮问数、追问、切换数据集                         |
| A1.4 Analysis Context           | 保存当前分析目标、数据集、知识、用户偏好      | ChatReport 尤其依赖这个能力                            |
| A1.5 User Preference            | 用户常用数据集、分析口径、图表偏好            | 个性化推荐问题和分析框架                               |
| A1.6 App Context Handoff        | 应用间上下文传递                              | ChatExcel 结果加入 ChatReport；ChatBI 图表加入报告素材 |
| A1.7 Notification / Task Center | 异步任务、学习任务、报告生成任务通知          | 后续 Deep Research / Report Agent 会需要               |

## A2. Data Center：数据接入与数据资产中心

```text
Data Center
= Data Source + Data Asset + Data Catalog + Metadata + Profile + Data Service
```

| L2 能力                 | 定义                                             | 说明                            |
| ----------------------- | ------------------------------------------------ | ------------------------------- |
| A2.1 数据源连接管理     | 连接 DB、DW、Lakehouse、API、对象存储、Excel/CSV | 复用 DB-GPT 已有数据源连接能力  |
| A2.2 文件数据接入       | Excel、CSV、压缩包、多 Sheet 文件上传与解析      | ChatExcel 的基础能力            |
| A2.3 数据源凭证管理     | 管理连接串、密钥、Token、权限委托                | 必须与安全模块联动              |
| A2.4 数据资产注册       | 将文件、表、视图、BI 数据集注册为 Dataset        | 后续所有应用都围绕 Dataset 消费 |
| A2.5 数据目录 / Catalog | 数据源、库、表、字段、Owner、标签、描述          | 让用户和 Agent 都能发现数据     |
| A2.6 技术元数据采集     | 表结构、字段类型、主键、索引、分区、刷新时间     | 支撑问数和质量判断              |
| A2.7 业务元数据维护     | 业务名称、字段解释、数据负责人、所属主题域       | 支撑业务理解                    |
| A2.8 数据预览           | 样例数据、TopN、空值、范围、分布                 | 给用户和 LLM 都看               |
| A2.9 数据服务接口       | 对外提供 Dataset API、Metadata API、Profile API  | ChatBI、ChatReport 都调用它     |
| A2.10 数据生命周期      | 上传、启用、冻结、归档、删除、版本               | 文件资产化必须具备              |
| A2.11 数据血缘占位      | 表、字段、SQL、报告之间的来源关系                | MVP 可先轻量，P1/P2 增强        |
| A2.12 数据服务市场占位  | 未来把高价值数据集发布为 Data Product            | 对应企业数据资产运营            |

## A3. Dataset Learning Center：数据集学习中心

这是当前阶段最关键的平台能力。

```text
Dataset Learning Center
= Profile + Field Semantics + Data Quality + Question Templates + Learning Job + Feedback Loop
```

| L2 能力                            | 定义                                     | 说明                                              |
| ---------------------------------- | ---------------------------------------- | ------------------------------------------------- |
| A3.1 Learning Job                  | 对数据集发起学习任务                     | 支持文件、表、BI 数据集                           |
| A3.2 Schema Understanding          | 理解表、Sheet、字段、类型、主外键候选    | DB/Excel 都需要                                   |
| A3.3 Field Semantics               | 字段语义识别                             | 如时间字段、金额字段、地区字段、ID 字段、分类字段 |
| A3.4 Field Alias / Synonym         | 字段别名、同义词、业务叫法               | “销售额”“GMV”“成交金额”映射                 |
| A3.5 Measure / Dimension Detection | 自动识别指标字段和维度字段               | ChatBI 问数准确率关键                             |
| A3.6 Data Profiling                | 空值率、唯一值、TopN、枚举值、最大最小值 | 支持维值匹配和质量判断                            |
| A3.7 Data Quality Scoring          | 数据质量评分                             | 缺失、重复、异常值、类型不一致                    |
| A3.8 Askable Field Scope           | 字段是否可问、是否可展示、是否敏感       | 与安全治理联动                                    |
| A3.9 Question Template Generation  | 自动生成推荐问题、快捷问题               | ChatExcel / ChatBI 共用                           |
| A3.10 Manual Correction            | 用户手动调整字段解释、同义词、可问范围   | Human-in-the-loop，提升准确率                     |
| A3.11 Learning Version             | 学习结果版本管理                         | 防止字段解释被误改                                |
| A3.12 Learning Feedback Loop       | 从成功/失败问数中回流学习资产            | QueryRun → Dataset Learning                      |
| A3.13 Dataset Card                 | 数据集说明卡                             | 面向用户展示数据集用途、字段、质量、样例问题      |
| A3.14 Cross-dataset Candidate      | 多数据集候选推荐                         | P1/P2 能力，不建议 MVP 做重                       |

当前附件里已经把 Dataset、DatasetField、DatasetProfile、QuestionTemplate、PersonalKnowledge、QueryRun 作为统一对象，这是非常正确的方向。

## A4. Semantic & Metric Center：语义层与指标中心

```text
Semantic & Metric Center
= Business Glossary + Metrics + Dimensions + Calculation Rules + Default Filters
```

| L2 能力                           | 定义                             | 说明                                     |
| --------------------------------- | -------------------------------- | ---------------------------------------- |
| A4.1 Business Glossary            | 业务术语表                       | 客户、活跃客户、有效线索、留存等         |
| A4.2 Metric Catalog               | 指标目录                         | 销售额、毛利率、转化率、ARPU             |
| A4.3 Metric Formula               | 指标计算公式                     | sum(amount)、count(distinct customer_id) |
| A4.4 Metric Grain                 | 指标粒度                         | 日、月、客户、门店、区域、产品           |
| A4.5 Dimension Catalog            | 维度目录                         | 时间、区域、渠道、产品、客户分层         |
| A4.6 Default Filters              | 默认过滤条件                     | 剔除测试数据、剔除退货、仅统计有效订单   |
| A4.7 Time Semantics               | 时间口径                         | 自然月、财务月、滚动 12 月、同比、环比   |
| A4.8 Metric Ownership             | 指标负责人和审批状态             | 企业级可信问数必需                       |
| A4.9 Metric Version               | 指标版本                         | 指标口径变化可追溯                       |
| A4.10 Field-to-Metric Mapping     | 字段到指标的映射                 | 从数据集学习结果升级为正式语义层         |
| A4.11 Semantic Query API          | 按语义层查询，而不是裸 SQL 查询  | 未来增强 ChatBI 准确率                   |
| A4.12 Semantic Conflict Detection | 发现同名不同口径、同义不同名问题 | 企业场景非常常见                         |

这里要明确：

> **Dataset Learning 是“理解数据”；Semantic & Metric Center 是“约束数据怎么被正确使用”。**

MVP 阶段可以先做轻量语义：字段别名、指标口径文本、推荐问题。
P1/P2 再做真正指标平台级的语义层。

## A5. Knowledge Center：知识中心

```text
Knowledge Center
= Document Knowledge + Business Rules + Analysis Framework + RAG + Knowledge Governance
```

| L2 能力                  | 定义                                         | 说明                                     |
| ------------------------ | -------------------------------------------- | ---------------------------------------- |
| A5.1 文档上传            | Word、PDF、飞书、Wiki、Markdown、历史报告    | 当前 MVP 可先 Word / 文本                |
| A5.2 文档解析            | 标题、段落、表格、图片说明、结构化抽取       | 报告类知识很依赖结构                     |
| A5.3 Chunking & Indexing | 切片、Embedding、索引                        | RAG 基础能力                             |
| A5.4 Knowledge Metadata  | 知识标题、来源、Owner、标签、适用范围        | 决定检索质量                             |
| A5.5 Knowledge Binding   | 绑定到用户、会话、数据集、主题域、应用       | 附件中建议第一阶段先个人级绑定，是合理的 |
| A5.6 指标口径知识        | 存放非结构化指标解释                         | 可逐步升级到 Metric Center               |
| A5.7 字段解释知识        | 对字段含义做补充说明                         | 与 Dataset Learning 联动                 |
| A5.8 分析框架库          | SWOT、漏斗分析、经营分析框架、财务分析框架等 | 支撑 ChatReport / Skill Hub              |
| A5.9 业务规则库          | 规则、例外、注意事项                         | 如“促销期需单独剔除异常流量”           |
| A5.10 检索策略           | 向量检索、关键词检索、Hybrid Search、Rerank  | 决定知识命中质量                         |
| A5.11 引用与溯源         | 回答中显示参考了哪些知识片段                 | 报告场景必须要有                         |
| A5.12 知识权限           | 个人、团队、部门、企业级可见范围             | MVP 先个人级，后续领域级/企业级          |
| A5.13 知识审批发布       | 领域知识发布、审核、版本                     | P1/P2 能力                               |
| A5.14 知识质量评分       | 命中率、使用率、用户反馈                     | 运营知识库必须具备                       |

建议第一阶段的定义保持克制：

```text
Personal Analysis Knowledge
= 用户上传的个人级分析上下文补充
≠ 企业级知识治理平台
```

## A6. Agent Runtime & LLM Adapter：Agent 运行时与模型适配层

这是 Data Agent 区别于传统 BI / 数据中台的核心。

```text
Agent Runtime
= LLM Adapter + Planner + Router + Memory + Tool Calling + Guardrails + Multi-Agent Orchestration
```

| L2 能力                       | 定义                                           | 说明                                       |
| ----------------------------- | ---------------------------------------------- | ------------------------------------------ |
| A6.1 LLM Provider Adapter     | 适配不同模型供应商                             | OpenAI、通义、Claude、DeepSeek、本地模型等 |
| A6.2 Model Router             | 按任务选择模型                                 | 问数、报告、代码、摘要可用不同模型         |
| A6.3 Prompt Template Center   | Prompt 模板管理                                | 问数、解释、报告、洞察各有模板             |
| A6.4 Context Builder          | 组装数据集、语义、知识、历史对话上下文         | 准确率关键                                 |
| A6.5 Intent Router            | 判断用户意图                                   | 问数、解释、画图、写报告、查知识、配置语义 |
| A6.6 Dataset Router           | 根据问题选择数据集                             | ChatBI / ChatDB 核心能力                   |
| A6.7 Planner                  | 把复杂问题拆成多步任务                         | ChatReport、Deep Analysis 必需             |
| A6.8 Tool Calling             | 调用 SQL、Python、Chart、Search、BI API 等工具 | Agent 行动能力                             |
| A6.9 Multi-Agent Handoff      | 在不同专家 Agent 间交接                        | 如问数 Agent → 洞察 Agent → 报告 Agent   |
| A6.10 Memory                  | 会话记忆、用户偏好、项目记忆                   | 不能只靠 prompt 临时拼接                   |
| A6.11 Structured Output       | 结构化输出 JSON / Schema                       | 便于渲染图表、报告、卡片                   |
| A6.12 Guardrails              | 输入、工具、输出的安全护栏                     | 防止越权 SQL、幻觉、敏感输出               |
| A6.13 Human-in-the-loop       | 需要用户确认时暂停                             | 指标口径不清、报告大纲确认、危险操作确认   |
| A6.14 Long-running Task State | 长任务状态保存、恢复、重试                     | ChatReport / Deep Research 必需            |

现代 Agent 工程已经越来越强调工具调用、handoff、guardrails、structured output、tracing 等运行时能力；OpenAI Agents SDK 官方文档也把 Agent 定义为带 instructions、tools、handoffs、guardrails、structured outputs 等运行行为的 LLM 应用构件。([OpenAI](https://openai.github.io/openai-agents-python/agents/?utm_source=chatgpt.com "OpenAI Agents SDK"))
对于复杂长任务，LangGraph 这类框架强调 durable execution、streaming、human-in-the-loop、persistence，这对 ChatReport、深度分析、长链路报告生成尤其关键。([LangChain 文档](https://docs.langchain.com/oss/python/langgraph/overview?utm_source=chatgpt.com "LangGraph overview - Docs by LangChain"))

## A7. Tool & Execution Hub：工具与执行中心

```text
Tool & Execution Hub
= SQL Executor + Python Sandbox + DuckDB + Chart Engine + Report Renderer + Tool Registry
```

| L2 能力                      | 定义                                  | 说明                                        |
| ---------------------------- | ------------------------------------- | ------------------------------------------- |
| A7.1 Tool Registry           | 工具注册中心                          | SQL、Python、Chart、BI API、Search 等       |
| A7.2 SQL Executor            | SQL 执行器                            | DB、DW、DuckDB、BI Dataset                  |
| A7.3 SQL Parser / AST        | SQL 解析与检查                        | 安全、审计、解释、重写                      |
| A7.4 DuckDB Runtime          | 文件/临时数据分析执行引擎             | ChatExcel 核心                              |
| A7.5 Python Sandbox          | 执行统计、建模、绘图、数据处理代码    | P1/P2 能力，需安全隔离                      |
| A7.6 Chart Engine            | 图表生成、图表配置、图表推荐          | ChatExcel / ChatBI / Dashboard 共用         |
| A7.7 Insight Algorithm Tools | 异常检测、贡献度、归因、聚类、预测    | 不应完全依赖 LLM 文本总结                   |
| A7.8 Report Renderer         | Word / PPT / PDF / Markdown 渲染      | ChatReport 必需                             |
| A7.9 Cache Layer             | 查询结果缓存、上下文缓存、Prompt 缓存 | 降成本、提性能                              |
| A7.10 Tool Permission        | 工具级权限                            | 谁能执行 SQL、谁能跑 Python、谁能访问某 API |
| A7.11 Tool Error Handling    | 工具失败解释、重试、降级              | 生产化必须有                                |
| A7.12 Tool Observability     | 每次工具调用的输入、输出、耗时、错误  | 进入 Run Trace                              |

OpenAI Agents SDK 的工具文档也强调工具可以用于取数、运行代码、调用外部 API，甚至将 Agent 作为工具调用；这类思想可以对应到我们的 Tool & Execution Hub。([OpenAI](https://openai.github.io/openai-agents-python/tools/?utm_source=chatgpt.com "Tools - OpenAI Agents SDK"))

## A8. Skill Hub：技能中心

```text
Skill Hub
= Analysis Skill Authoring + Skill Packaging + Skill Publishing + Skill Execution + Skill Evaluation
```

| L2 能力                    | 定义                                    | 说明                                         |
| -------------------------- | --------------------------------------- | -------------------------------------------- |
| A8.1 Skill Definition      | 定义一个分析技能是什么                  | 输入、步骤、工具、输出、适用场景             |
| A8.2 Skill Template        | 官方技能模板                            | 趋势分析、同比环比、漏斗、归因、经营分析     |
| A8.3 Skill Extraction      | 从历史分析过程一键沉淀技能              | 你提到的“从历史 data application 导出技能” |
| A8.4 Skill Authoring       | 用户/官方编辑技能                       | 可视化或配置化                               |
| A8.5 Skill Version         | 技能版本管理                            | 防止分析逻辑不可追溯                         |
| A8.6 Skill Test            | 用样例数据和问题测试技能                | 类似技能单元测试                             |
| A8.7 Skill Publish         | 个人、团队、官方发布                    | 对应技能市场                                 |
| A8.8 Skill Invocation      | 在 ChatExcel、ChatBI、Report 中调用技能 | 技能成为 Agent 工具                          |
| A8.9 Skill Evaluation      | 成功率、准确率、使用率、满意度          | 技能运营                                     |
| A8.10 Skill Governance     | 谁能发布官方技能，谁能修改              | 企业级必须有                                 |
| A8.11 Skill Dependency     | 技能依赖的数据集、指标、知识、工具      | 便于迁移和审计                               |
| A8.12 Skill Recommendation | 根据场景推荐技能                        | 如“这是漏斗分析问题，推荐漏斗分析技能”     |

建议把 Skill 定义成：

> **可复用、可执行、可评估的分析方法单元。**

不要只把 Skill 做成 Prompt 模板，否则价值太浅。

## A9. Asset Hub / Artifact Hall：资产与成果大厅

```text
Asset Hub
= Dataset Asset + Analysis Artifact + Insight Card + Report Material + Case Library + Reusable Framework
```

| L2 能力                       | 定义                             | 说明                                     |
| ----------------------------- | -------------------------------- | ---------------------------------------- |
| A9.1 Dataset Asset            | 数据集资产                       | 文件、表、BI 数据集、临时分析数据集      |
| A9.2 Chart Asset              | 图表资产                         | 从 ChatExcel、ChatBI、Dashboard 生成     |
| A9.3 Table Result Asset       | 表格结果资产                     | 可加入报告、二次分析                     |
| A9.4 SQL Asset                | 成功 SQL、模板 SQL               | 可复用、可审计                           |
| A9.5 Insight Card             | 洞察卡片                         | 趋势、异常、归因、建议                   |
| A9.6 Report Material          | 报告素材                         | 图表、结论、数据引用、说明文字           |
| A9.7 Report Asset             | 完整报告                         | Word、PPT、Markdown、PDF                 |
| A9.8 Case Library             | 官方分析案例                     | 行业案例、部门案例、经营分析案例         |
| A9.9 Analysis Framework Asset | 分析框架                         | 经营分析框架、财务分析框架、销售漏斗框架 |
| A9.10 Learning Artifact       | 数据集学习产物                   | Dataset Card、字段解释、推荐问题         |
| A9.11 Artifact Lineage        | 资产血缘                         | 这个图表来自哪个数据集、SQL、问题、知识  |
| A9.12 Asset Search            | 搜索和推荐资产                   | 搜图表、搜报告、搜案例                   |
| A9.13 Asset Reuse             | 一键复用到新分析、新报告、新看板 | 平台价值的关键                           |
| A9.14 Asset Governance        | 收藏、分享、发布、下架、权限     | 资产运营必需                             |

Asset Hub 的核心不是“文件仓库”，而是：

> **把一次性分析结果沉淀成可复用的企业分析资产。**

## A10. Governance, Permission & Security：治理、权限与安全

```text
Governance & Security
= IAM + Data Permission + SQL Guard + Privacy + Audit + Compliance
```

| L2 能力                         | 定义                                      | 说明                               |
| ------------------------------- | ----------------------------------------- | ---------------------------------- |
| A10.1 IAM / SSO                 | 统一认证                                  | 企业接入必需                       |
| A10.2 RBAC                      | 基于角色授权                              | 管理员、分析师、业务用户           |

| A10.13 Compliance Policy        | 合规策略                                  | 金融、隐私、跨境、企业内控         |

# 18. 最重要的 5 个架构判断

最后我提炼成 5 个判断，作为后续讨论的锚点。

## 判断 1：Data Agent 的底座不是 LLM，而是“数据上下文工程”

LLM 是推理和交互层，但企业级 Data Agent 的准确率来自：

```text
数据资产
+ 字段语义
+ 指标口径
+ 知识上下文
+ 权限边界
+ 运行反馈
```

## 判断 2：Dataset Learning 是平台核心，不是 ChatExcel 功能

ChatExcel 是第一消费场景，但 Dataset Learning 必须平台化。
否则 ChatBI、ChatDB、ChatReport 后面都会重复建设。

## 判断 3：Knowledge Center 和 Semantic Center 要分开

Knowledge Center 管非结构化知识。
Semantic Center 管可执行语义和指标口径。
二者可以互相转化，但不能混为一个模块。

## 判断 4：Skill Hub 是 Data Agent 的长期护城河

Text2SQL 会越来越通用，但企业自己的分析技能、业务框架、报告套路、行业方法论才是差异化资产。

## 判断 5：Run Trace & Eval 必须从第一天做

没有运行留痕，就没有：

```text
准确率评估
失败问题复盘
Prompt 优化
模型切换
技能沉淀
知识改进
数据学习闭环
```

所以它不是“后期运维功能”，而是 Data Agent 的产品核心闭环。
