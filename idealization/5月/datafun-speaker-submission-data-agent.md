## 演讲过程设计

建议按“黄金结构”提交，可作为附件内容基础。

### 1. 背景引入：为什么需要 Data Agent

- 企业数据分析需求正在从固定报表走向自然语言问数、解释、归因和报告生成。
- 传统链路通常是“业务提需求 -> 数据团队取数/做报表 -> 业务继续追问”，响应周期长，分析经验难沉淀。
- 大模型带来了自然语言交互能力，但直接让模型猜表、猜字段、猜指标口径并不可靠。

### 2. 核心判断：Data Agent 不是聊天框，而是分析基础设施

- Data Agent 的关键不只是模型能力，而是 AI-ready 数据资产和可信运行体系。
- 需要把个人文件、数仓宽表、语义数据集、指标、看板、报告和业务知识统一成可理解、可调用、可追踪的对象。
- 企业级落地要同时解决可问范围、指标口径、SQL 安全、权限审计、证据链和反馈评测。

### 3. 场景地图：从个人探索到企业经营

- 个人文件分析：Excel/CSV 上传、字段探索、自然语言问数、质量洞察。
- 领域数据集分析：语义数据集问数、多维下钻、异常发现。
- 指标分析：指标解释、同比环比、归因分析、指标预警。
- 报告分析：经营月报、专题复盘、自然语言生成报告、智能周报/月报。

### 4. 架构拆解：Data Agent 应该处在什么位置

- 向上服务 T-Claw/PI-Claw、ChatBI、小Q式入口、FineBI、报告生成入口。
- 中间负责统一编排、语义注入、安全运行、产物沉淀和评测运营。
- 向下连接数仓、指标平台、知识库、BI 看板、文件和数据库。
- FineBI 负责稳定报表，ChatBI 负责自然语言入口，Data Agent 负责深度分析和报告编排。

### 5. 实践探索：几个关键能力闭环

- 可信问数：DataSet 语义层、字段别名、样例问题、指标口径、NL-to-SQL、SQL 安全网关。
- 看板解读：图表元数据、筛选上下文、底层数据和业务口径共同进入解读流程。
- 报告生成：模板驱动 + LLM 辅助生成，保存章节、图表、SQL、引用关系和人工反馈。
- Skill Hub：将问数、解读、归因、报告封装为可复用能力包，而不是一次性 Prompt。

### 6. DB-GPT 适配边界

- 可复用：多数据源连接、Chat Scene、LLM 调用、Excel/CSV 分析、SQL 执行、报告原型。
- 需补齐：数据集配置层、指标语义、SQL 校验、AnalysisChain、ReportRun、Artifact、Eval。
- 定位：DB-GPT 更适合作为开放可控的 Data Agent Runtime，而不是完整 BI 或小Q替代品。

### 7. 落地挑战与解决思路

- NL-to-SQL 准确率：通过黄金数据集、业务知识库、Few-shot、SQL 校验和回归评测提升。
- 数据安全：只读 SQL、限行、超时、敏感字段脱敏、行列权限和审计日志。
- 报告可信：不做纯文本总结，每段结论保留数据、SQL、图表和指标口径引用。
- 成本与时延：Stage 并行、缓存、流式输出和模型路由。

### 8. 展望：从问数 Agent 到分析 Agent

- 短期：先跑通文件问数、DB 可信问数、看板解读和经营报告生成。
- 中期：建设黄金数据集、指标知识库、Skill Hub 和反馈评测体系。
- 长期：从被动问答走向主动洞察、归因、任务生成和经营协同。

## 听众收益

1. 理解企业级 Data Agent 与传统 BI、ChatBI、小Q式入口的定位差异，避免把智能分析简单做成“聊天框”。
2. 获得一套可复用的建设框架：数据对象、分析自由度、治理等级、语义层、运行态、产物沉淀和评测反馈。
3. 了解可信问数、看板解读、经营报告生成在企业落地时的关键挑战与工程化解决思路。

## 落地挑战和方案重点

1. **可信问数难**：大模型直接生成 SQL 容易误解字段和指标口径。方案重点是建设语义数据集、业务术语、样例问题、SQL 安全网关和回归评测集。
2. **分析结果难复用**：一次性问答无法形成企业资产。方案重点是沉淀 AskRun、InsightRun、ReportRun、Artifact、报告模板和分析 Skill。
3. **系统边界难划清**：Data Agent 容易和 BI、ChatBI、数仓重复建设。方案重点是明确分工：BI 做稳定展示，ChatBI 做自然语言入口，Data Agent 做分析编排、证据链和报告生成。
