# 一、背景

## 1.1 业务侧涌现的真实需求

在实际业务场景中，大家对数据能力的需求已经不只是“看一个已有看板”，而是越来越多地希望通过更自然、更灵活的方式完成取数、分析和后续动作。

### **问数：**当需要某个指标的数据时，希望可以通过对话的形式，快速便捷地查询出来

1. 可以帮忙接入我们制造领域的数据吗？
2. 我想使用问数功能，权限要怎么申请？
3. 我需要在 xx 业务系统中新增问数场景，想问下要如何实现？

### **分析报告：**基于经营、运营数据的数据总结、提炼，进一步到分析、洞察、归因等层面

1. 本月的月度经营情况汇总汇报
2. 某个门店的近期经营异常分析，以及督导巡店改进意见
3. 为什么上个月的毛利跌了？是产品力问题，是市场需求问题，是渠道效率问题，还是成本问题？那我可以做什么？调价、让利，还是保持战略方向不变？

这些问题背后，本质上是业务侧对数据能力的要求在变化：从过去“看数据”，逐步走向“问数据”，再进一步走向“围绕一个业务问题完成分析、报告和行动跟踪”。

## 1.2 Data+AI 的探索

如果从数据能力的演进来看，可以大致分成三个阶段。

### **1.0：Dashboard 阶段**

关键词是：标准化、可视化、中心化交付。

这个阶段，大数据部门主要提供 BI 看板和报表。对于标准指标、固定分析口径、高频经营场景，这种方式比较有效，可以保证数据口径统一、展示稳定、交付集中。

但它也有一个明显限制：当业务问题变得更临时、更细碎，或者需要跨领域分析时，单靠固定看板很难覆盖所有问题。

### **2.0：ChatBI 阶段**

关键词是：自然语言、自助问数、智能解读。

这个阶段，数据使用方式开始从“看数据”走向“问数据”。用户可以通过自然语言提出问题，系统帮助理解指标、维度、筛选条件，并返回查询结果或初步解读。

团队内部早期也做过一些相关探索，例如 DataClaw、ReportClaw，以及基于 QuickBI 的智能小Q等，主要解决的是用户取数、数据解读和初步分析的问题。

### **3.0：Data Agent / 共创式分析阶段**

关键词是：业务共创、AI 生成、工程化交付、业务闭环。

这个阶段不只是回答一个数据问题，而是希望围绕一个业务场景，进一步生成分析结论、报告产物，甚至是可以复用的看板或分析应用。

在这个过程中，超级用户可以通过 vibe coding 的方式生成看板或分析应用；IT / 大数据部门则提供数据接入、语义口径、权限、部署和工程化保障。

## 1.3 今天分享的 TClaw 实践

今天分享的 TClaw 数据分析实践，严格来说还不是完整的 3.0。它更像是从 2.0 ChatBI 往 3.0 Data Agent / 共创式分析过渡中的一个典型实践。

它已经不只是简单问数，而是开始具备一些更完整的分析能力，例如：

1. 根据用户问题查询对应指标数据
2. 结合已有数据进行分析
3. 读取飞书多维表格
4. 生成 Html 报告或飞书云文档报告
5. 根据分析结果做任务跟踪

所以这次分享的重点，不只是介绍一个问数工具，而是通过 TClaw 的实际案例，看一下 Data+AI 如何在真实业务场景中，从“回答数据问题”进一步走向“生成分析产物”和“推动业务闭环”。
