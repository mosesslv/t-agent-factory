## 建议结构

### 一、背景：从看数据、问数据，到共创式分析

#### 1.1 业务需求正在从“取数”走向“分析闭环”

保留现有真实需求，但不要一开始就散列问题。建议先总结需求变化：

- 第一类是取数效率：业务希望用对话方式快速查询指标，不再反复找报表、找口径、找权限入口。
- 第二类是数据接入：制造、零售、财务、产销等领域希望把自己的业务数据接入问数或分析场景。
- 第三类是经营分析：业务不只要知道“数是多少”，还希望理解异常原因、行动建议和后续跟踪。

可继续使用现有例子：

- 可以帮忙接入制造领域的数据吗？
- 我想使用问数功能，权限要怎么申请？
- 我需要在业务系统中新增问数场景，要如何实现？
- 为什么上个月毛利跌了？是产品力、市场需求、渠道效率还是成本问题？下一步应该调价、让利，还是保持战略方向？

#### 1.2 数据交付能力经历三个阶段

这一节用于承接你补充的三阶段思考。

**1.0 Dashboard 阶段：标准化、可视化、中心化交付**

这个阶段，大数据部门主要通过 BI 看板和固定报表交付数据能力。它适合标准指标、稳定流程和高频复用场景，优势是口径统一、展示稳定、管理可控；但当业务问题变得临时、细碎、跨域时，单靠看板很难覆盖。

**2.0 ChatBI 阶段：自然语言、自助问数、智能解读**

这个阶段，能力从“看数据”走向“问数据”。业务可以用自然语言提问，系统自动理解指标、维度、筛选条件，并返回查询结果或初步解读。团队内部探索的 ChatBI、DataClaw、ReportClaw、QuickBI 智能小Q，都可以放在这个阶段下介绍。

**3.0 Data Agent / 共创式分析阶段：业务共创、AI 生成、工程化交付、业务闭环**

这个阶段，目标不只是回答一个数据问题，而是让业务和技术共同完成一个分析产物或轻量应用。超级用户可以通过 vibe coding 生成看板、分析页面或报告应用；IT / 大数据部门则提供数据接入、语义口径、权限控制、部署发布和工程化保障。

#### 1.3 TClaw 实践的位置：从 2.0 走向 3.0 的过渡样本

建议把转折讲清楚：

今天分享的 TClaw 数据分析实践，严格来说还不是完整的 3.0。它仍然依赖既有的数据接口、语义能力和工具链，也还没有完全形成面向业务共创的产品化闭环。

但它已经不是传统意义上的问数工具。它开始具备几类更接近 Data Agent 的能力：

- 能围绕业务问题做分析，而不只是返回一个指标值。
- 能生成 HTML 或飞书云文档形式的报告。
- 能读取飞书多维表格，把业务过程数据纳入分析。
- 能结合任务跟踪，把分析结果继续推进到执行闭环。

因此，这次分享可以定位为：一次从 ChatBI 能力向 Data Agent / 共创式分析能力过渡的实践复盘。

## 可直接替换的一版行文

### 一、背景

过去，大数据部门的数据服务更多围绕 BI 看板和固定报表展开。对于标准指标、稳定流程和高频经营场景，这种方式能够保证口径统一、展示稳定、集中交付，我们可以把它理解为 Data + AI 之前的数据交付 1.0 阶段：Dashboard 阶段。

但随着业务经营越来越精细，业务侧提出的问题也在发生变化。大家不只是想“看一个已有看板”，而是希望在具体场景里快速获得答案。例如：某个指标的数据能不能通过对话直接查出来？制造领域的数据能不能接入问数？业务系统里能不能新增一个问数场景？当月经营情况能不能自动汇总？某个门店经营异常，能不能进一步分析原因并给出巡店改进建议？甚至，当毛利下跌时，业务真正想知道的是：问题来自产品力、市场需求、渠道效率还是成本？下一步应该调价、让利，还是坚持原有策略？

这些需求背后，其实是数据能力从 1.0 向 2.0、3.0 演进的信号。

2.0 阶段可以称为 ChatBI 阶段，关键词是自然语言、自助问数和智能解读。它让数据使用方式从“看数据”走向“问数据”：业务用户可以用自然语言提出问题，系统帮助理解指标、维度和筛选条件，并返回查询结果或初步分析。团队内部已经在这个方向做了不少探索，包括 ChatBI、DataClaw、ReportClaw，以及 QuickBI 的智能小Q。

但业务需求还在继续往前走。真正有价值的场景，往往不止于问出一个数，而是要形成一个分析结论、一个报告产物、一个可跟踪的行动方案，甚至是一个可以被复用的轻量分析应用。这对应的是 3.0 阶段：Data Agent / 共创式分析阶段。这个阶段的关键词是业务共创、AI 生成、工程化交付和业务闭环。超级用户可以通过 vibe coding 生成看板或分析应用；IT / 大数据部门则负责数据接入、语义口径、权限控制、部署发布和工程化保障。

今天分享的 TClaw 数据分析实践，严格来说还不是完整的 3.0。它更像是从 2.0 ChatBI 向 3.0 Data Agent 过渡中的一个典型实践：它已经不只是问数，而是开始具备分析、生成报告、读取多维表格、任务跟踪等能力。也正因为如此，我们可以通过这个案例，看看 Data + AI 在真实业务分析场景里，如何从“回答问题”进一步走向“交付分析结果”和“推动业务闭环”。
