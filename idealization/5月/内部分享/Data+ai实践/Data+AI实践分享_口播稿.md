# Data+AI 实践分享口播稿

> 建议总时长：13-14 分钟  
> 节奏：背景 3 分钟，案例 7-8 分钟，思考和下一步 3 分钟  
> 使用方式：飞书文档负责展示结构、截图和关键判断；这份稿子负责口播逻辑，不需要逐字读飞书文档。

## 开场 20 秒

大家好，今天这次分享主要不是介绍一个新的问数工具，而是结合我们最近在 TClaw 上的数据分析实践，聊一下 Data+AI 在企业内部数据场景里，正在从“看数据”“问数据”，进一步走向“分析产物共创”和“业务闭环”。

我会分三部分讲：第一，为什么业务侧的数据需求在变化，以及我们大数据部门过去做过哪些探索；第二，TClaw 现在具体能做到什么；第三，使用 AI 做数据分析时要注意什么，以及后续我们会怎么继续补能力。

## 一、背景 3 分钟

先看背景。

最近业务侧对数据能力的需求，已经不只是“帮我做一个看板”或者“我去看一个已有报表”。我们看到的需求大概可以分成两类。

第一类是问数。比如业务同事希望接入制造、零售、财务等领域的数据后，可以直接用自然语言问：“这个指标是多少？”“某个国家最近趋势怎么样？”过去这些问题可能要找报表、找人取数，或者自己去系统里筛选，现在大家希望可以用更低门槛的方式快速拿到结果。

第二类是分析报告。这个需求比问数更进一步。业务不是只想知道一个数，而是希望 AI 能围绕一个经营问题做总结、归因和建议。比如月度经营情况怎么总结，某个门店异常怎么分析，毛利为什么下降，后面应该调价、让利，还是继续保持原策略。

这些需求背后，本质上是企业对“数据驱动业务”的需求在升级。过去更多是人去看数据，而且这个人要会看；现在开始希望更快拿到有用的信息；再往后，会希望围绕具体业务场景和一系列问题，完成数据分析、报告撰写和行动建议。

面对这些变化，大数据部门不是从 TClaw 才开始探索。我们的能力供给本身也经历了一个演进过程。

最早是 BI / QuickBI 看板，主要解决标准指标和固定报表问题。这类方式适合高频、稳定、有统一口径的经营场景。

后来是 ChatBI 问数，主要解决自然语言查数的问题。用户不一定要进入复杂报表，可以直接问一个指标、一个趋势，系统返回结果。

再往后，我们又围绕 QuickBI 智能小Q、DataClaw、ReportClaw 做了一些探索，开始处理智能解读、复杂意图理解、分步骤分析和自动生成报告。这些探索也沉淀了一些数据分析相关的 skills。

这条探索旅程背后，其实对应的是数据分析范式的变化。

过去更多是平台统一建设看板，业务查看结果，这是“看数据”。后来开始让业务用自然语言问数据，这是“问数据”。再往后，会有一批超级用户基于自己的场景和数据生成分析产物，大数据部门则补上数据接入、语义口径、权限和工程化能力。

所以我们今天讲 TClaw，不是因为它只是一个新的问数入口，而是因为前面这些探索沉淀下来的 skills 正在逐步接入 TClaw。TClaw 开始把问数、分析、报告、看板、分享和任务跟踪串成一条链路。

下面进入案例部分，看它具体体现出了哪些能力。

## 二、Data+AI 案例介绍 7-8 分钟

第二部分重点看 TClaw 的实践案例。

先给大家一个整体印象：这组能力不是单点问数，而是覆盖了“提问 - 分析 - 生成报告 - 分享成网页链接 - 跟踪行动”的完整链路。

也就是说，它不仅能回答“这个数是多少”，还可以进一步完成趋势分析、读取用户自己的多维表格、生成 HTML 报告或飞书云文档报告，并根据分析结果触发任务跟踪。

前置准备这里简单带过。要使用这些能力，先需要在 TClaw 里创建分析子 Agent，并安装必要技能，比如 smart-data-agent、htmlnow、飞书 CLI 等。这里不是今天重点，大家只需要理解：TClaw 的能力不是一个孤立入口，而是通过这些 skills 把问数、报告、飞书能力串起来。

接下来看基础问数。

第一个案例是类似“尼日本月激活是多少”这样的基础问题。用户提出问题后，TClaw 不是直接给一个结果，而是先理解问题，返回查询思路给用户确认。确认无误后，再执行查询并返回结果。这个环节主要验证的是取数准确性，以及它能不能把自然语言问题转成明确的数据查询。

第二个案例是复杂一点的查询，比如“孟加拉国的销量趋势如何”。这类问题不是简单查一个指标，而是需要先理解分析目标，再制定分析计划。用户可以继续修改问题、补充条件，TClaw 会重新生成分析思路，再执行分析。这个地方体现的是“可追问、可修正”，比传统的一次性查数更接近真实分析过程。

接下来是报告生成，这是我认为比较关键的一步。

过去很多 AI 分析停留在对话框里，最后给你一段文字。问题是这段文字很难沉淀，也不方便分享。TClaw 这里可以把分析结果整理成 HTML 报告，并生成一个可点击的在线链接。用户不需要只看聊天记录，而是可以打开一个结构化页面，看趋势、结论和图表。

这个能力的意义在于：分析结果开始从“回答”变成“产物”。一旦变成报告链接，就可以被分享、复用，也更适合团队协作。

再往后，如果某类分析需要周期性查看，就可以从单次问答式分析，进一步配置成定时推送。比如每周、每月固定生成某类分析结果，再推送给相关人员。这个能力让分析不只是一次性动作，而是可以变成持续触达。

最后看多维表格 + 飞书报告 + 任务跟踪这个闭环案例。

这个案例更接近我们前面说的共创式分析特征。用户不一定只使用大数据部门已经接好的标准数据，也可以基于自己手头的多维表格发起分析。TClaw 读取表格内容后，先给出分析思路，用户确认后，可以生成飞书云文档报告。

如果报告里有需要别人跟进的事项，还可以自动 @ 对应负责人，生成任务并发送提醒。

这个链路完整串起来后，就不只是“AI 帮我查个数”，而是从用户自己的数据开始，经过分析、报告沉淀，再到任务跟踪。它已经开始具备“分析产物 + 团队协作 + 执行闭环”的特点。

简单小结一下这一部分。

TClaw 当前比较适合几类场景：数据问答、趋势分析、报告生成、个人数据看板 DIY、多维表格分析、任务跟踪。当前建议超级用户可以先基于已接入的数据域试用。如果只是试用已有数据，可以按流程配置；如果要接入新的业务数据或新增问数场景，需要先确认数据源、指标口径和权限，再由 IT / 大数据部门协助接入。

目前零售域的数据 CLI 接口已经上线，财务域也在本月推进，后续产销域、营销域等会陆续上线。

## 三、一些思考和下一步 3 分钟

最后讲一些思考和下一步。

前面的案例看起来比较顺，但这里需要提醒一点：Data+AI 不是把数据直接丢给 AI，就一定能得到可直接用于经营判断的结论。

业务在用 AI 做数据分析时，最容易遇到三类问题。

第一是口径问题。比如同样是销量、激活、毛利，不同团队、不同系统、不同报表里可能定义不一样。如果没有说明清楚口径，AI 可能会按自己的理解去取数和分析，最后结果看起来合理，但不一定符合真实业务口径。

第二是计算问题。很多分析不是简单查一个数，而是要算同比、环比、占比、毛利率、达成率。这里分子分母、时间范围、聚合层级、过滤条件只要有一个不一致，结果就会偏。所以不能只看 AI 最后的文字结论，也要看它用了什么数据、怎么算出来的。

第三是业务背景问题。AI 可以基于已有数据做总结和归因，但它不一定知道所有业务背景。比如销量下降，可能和渠道、价格、库存、活动节奏、竞品动作、节假日、数据延迟都有关系。如果这些背景没有进入数据或问题描述，AI 的结论就可能不完整。

所以我们对 TClaw 的定位不是替代业务判断，而是帮助用户更快地把问题拆开，把数据查出来，把计算过程展示出来，把报告初稿生成出来。最终能不能用于经营判断，还需要业务侧确认口径、计算逻辑和业务背景。

接下来哪些场景可以先用起来？

我建议先从个人场景和小范围场景开始，不要一开始就追求完整团队级落地。比如基于已有数据域做问数、趋势分析、读取多维表格、生成 HTML 或飞书云文档报告。QuickBI 智能小Q更适合围绕已有数据集和已有看板做问数、解读和报告；TClaw 更适合用户基于自己的问题和手头数据，进一步生成分析产物。

从最近的使用情况看，我们也观察到一个趋势：会先出现一批愿意自己动手的超级用户。

这类用户熟悉业务问题，手上有自己的数据或分析场景，也愿意自己尝试。他们不满足于只问一个数，而是希望生成看板、报告或轻量分析应用，再分享给团队其他人看。

后面的协作方式可能会变成：前半段，业务侧和超级用户先跑起来，提出场景、准备数据、生成初版分析产物，小范围验证价值；后半段，IT / 大数据部门补工程化能力，包括数据接入、指标和语义口径沉淀、权限控制、后端服务、部署和稳定性。

也就是说，看板和分析应用可以先 DIY，但从 demo 到生产，仍然需要工程化能力来支撑。

下一步我们会围绕一个方向继续推进：把 Data+AI 能力做成更容易被业务使用，也更容易被工程化沉淀的基础能力。具体包括更多可复用的 skills，黄金 / 语义数据集 CLI，以及 T-agent、QuickBI 智能小Q 等平台产品能力。

最后总结一句：TClaw 今天展示的不是一个终点，而是一个过渡阶段的实践。它让我们看到，Data+AI 正在从“问数工具”走向“业务共创分析产物”。后续大数据侧要做的事情，就是把数据、语义、权限、部署和平台能力逐步补齐，帮助这些个人产物真正变成可以分享、复用、持续运行的团队能力。
